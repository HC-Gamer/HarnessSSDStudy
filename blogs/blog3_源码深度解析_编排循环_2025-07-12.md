# 读 OpenCode 源码：一个「没有步数限制的循环」如何撑起 Agent 的全部智能 (2025-07-12)

> 源码：`github.com/opencode-ai/opencode` | 语言：Go | 核心目录：`internal/llm/agent/`、`internal/llm/prompt/`

---

## 一、为什么要读 Agent 的源码

现在讲 AI Agent 的文章多如牛毛，但大多停留在两个层次：要么是概念图——「Planning、Memory、Tools、Reflection」四个方框加几根箭头；要么是框架 API 的搬运——教你怎么调 LangChain 的 `AgentExecutor`。这两类内容有个共同问题：**它们把 Agent 讲成了一个黑盒，而黑盒里最关键的那个「循环」到底长什么样，反而没人说清楚。**

我一直有个偏执的信念：任何一个系统，只要你不知道它的主循环（main loop）是怎么转的，你就没真正理解它。操作系统的调度循环、游戏引擎的 tick 循环、事件驱动服务器的 event loop——**主循环是一个系统的世界观**。Agent 也一样。所谓「智能体」，剥掉所有营销词汇，内核就是一个循环：观察、思考、行动、更新状态，然后再来一遍。

所以当我想搞清楚「一个 Agent 到底是怎么运行的」时，我没有去读某个重量级框架（那些框架的抽象层太多，反而看不清本质），而是选了 OpenCode。原因很简单：

1. **它是 Go 写的**。Go 没有 Python 那种魔法，没有装饰器嵌套、没有元类，控制流是白纸黑字的 `for`、`select`、`channel`，读起来一览无余。
2. **它足够小，但功能完整**。它是一个能真正干活的 CLI 编程助手——有工具调用、有子 Agent、有上下文压缩，该有的都有，但没有为了「通用」而堆砌的抽象。
3. **它的循环写得非常「诚实」**。读完之后你会发现，一个 production 级 Agent 的核心循环，真的就是一个 `for {}` 加一个 `continue`。

这篇文章我想带你一层层剥开这个循环。不是逐行翻译注释，而是去看每个设计决策背后的**意图**和**权衡**——为什么是这样，而不是那样。

## 二、架构总览：一个 channel 驱动的 OTAU 循环

先给一张全局图。OpenCode 的编排循环遵循经典的 **Observe → Think → Act → Update State**（我私下叫它 OTAU），整个流程实现为一个**基于 channel 的事件驱动流**：

```
                    ┌────────────────────┐
                    │    User Input      │
                    └────────┬───────────┘
                             │
                    ┌────────▼───────────┐
                    │  processGeneration │  ← 主循环入口
                    └────────┬───────────┘
                             │
              ┌──────────────▼──────────────┐
              │   streamAndHandleEvents     │  ← Observe + Think
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │   FinishReason == ToolUse?  │  ← Act 决策
              └──────┬──────────────┬───────┘
                     │ NO           │ YES
              ┌──────▼────┐  ┌──────▼──────────────┐
              │ 返回结果   │  │ 执行工具 → 追加历史  │
              │ Done=true  │  └──────┬───────────────┘
              └───────────┘         │
                     ┌──────────────▼──────────────┐
                     │      回到 for 循环顶部       │
                     └─────────────────────────────┘
```

这张图里最值得玩味的，是**那根从右下角绕回顶部的箭头**。它就是整个 Agent「智能感」的来源。我们后面会反复回到它。

先记住一个心智模型：**LLM 本身是无状态的、一次性的函数**——你给它一堆消息，它吐一段回复。它不会「持续运行」，不会「主动做事」。所谓 Agent 的自主性，完全是外面那个循环「骗」出来的：循环不停地把「上一轮做了什么 + 结果」喂回给这个无状态函数，让它误以为自己在连续地工作。**Agent 的智能不在模型里，而在循环里**——这是我读完这份源码最大的一个体感。

## 三、主循环：`processGeneration`

直接上核心，`agent.go:233-311`：

```go
func (a *agent) processGeneration(ctx context.Context, sessionID, content string, ...) AgentEvent {
    // 1. 读取历史消息
    msgs, err := a.messages.List(ctx, sessionID)

    // 2. 创建用户消息，追加到历史
    userMsg, _ := a.createUserMessage(ctx, sessionID, content, attachmentParts)
    msgHistory := append(msgs, userMsg)

    // 3. 主循环：观察 → 思考 → 行动 → 更新
    for {
        // 3a. 检查取消
        select {
        case <-ctx.Done():
            return a.err(ctx.Err())
        default:
        }

        // 3b. 流式调用 LLM，处理事件（Observe + Think）
        agentMessage, toolResults, err := a.streamAndHandleEvents(ctx, sessionID, msgHistory)

        // 3c. 如果 LLM 决定使用工具（Act）
        if (agentMessage.FinishReason() == message.FinishReasonToolUse) && toolResults != nil {
            // 追加助手消息 + 工具结果到历史（Update State）
            msgHistory = append(msgHistory, agentMessage, *toolResults)
            continue // ← 回到循环顶部！
        }

        // 3d. LLM 认为任务完成，返回结果
        return AgentEvent{Type: AgentEventTypeResponse, Message: agentMessage, Done: true}
    }
}
```

第一次读到这段代码时，我是有点「就这？」的失落感的——然后紧接着是一种更强烈的 aha moment。**一个能自主搜索代码库、写代码、跑测试、修 bug 的智能体，它的中枢就是一个裸的 `for {}`。**

这里有三个设计细节值得停下来品：

**其一，退出条件不是步数，而是 `FinishReason`。** 循环没有 `for i := 0; i < maxSteps; i++`，而是无限循环。什么时候停？当且仅当 LLM 返回的 `FinishReason` **不是** `ToolUse`——也就是模型自己认为「我不需要再调工具了，可以给最终答复了」。**决定循环何时终止的权力，被完全交给了模型自己。** 这是一个大胆的设计，后面单独讲。

**其二，状态的载体极其朴素——就是 `msgHistory` 这个 slice。** 没有复杂的状态机，没有 blackboard，没有 RAG 向量库掺和进主循环。Agent 的「记忆」「上下文」「进展」，统统就是这个不断 `append` 的消息数组。`Update State` 这一步，物理上就是一行 `append`。这种「历史即状态」的设计，和 LLM「拿全部上下文重新推理」的本质是严丝合缝的——每一轮循环，模型都是看着完整的历史从头思考一遍，所以你根本不需要额外维护什么中间状态，历史本身就是状态。

**其三，取消检查放在循环顶部的 `select`。** 因为一次工具调用可能很慢（跑测试、装依赖），用户随时可能 Ctrl-C，每一轮开头非阻塞地 `select { case <-ctx.Done() }` 是 Go 里最地道的协作式取消。小细节，但体现了这是给真人用的工具，不是玩具。

## 四、Observe + Think：`streamAndHandleEvents` 与流式事件

`processGeneration` 是骨架，真正的血肉在 `streamAndHandleEvents`（`agent.go:322-438`）。它把「调 LLM」和「执行工具」两件事缝在了一起：

```go
func (a *agent) streamAndHandleEvents(ctx context.Context, sessionID string, msgHistory []message.Message) (...) {
    // 1. 流式调用 LLM
    eventChan := a.provider.StreamResponse(ctx, msgHistory, a.tools)

    // 2. 逐个处理流事件
    for event := range eventChan {
        a.processEvent(ctx, sessionID, &assistantMsg, event)
    }

    // 3. LLM 流结束后，执行所有工具调用
    for i, toolCall := range toolCalls {
        toolResult, toolErr := tool.Run(ctx, tools.ToolCall{...})
        toolResults[i] = message.ToolResult{...}
    }

    return assistantMsg, &toolMsg, nil
}
```

注意这里用的是 `StreamResponse`，返回的是一个 `eventChan`——**这是整个设计里我最欣赏的一处工程决策**。为什么不直接一个 `SendMessages` 拿到完整回复，非要搞流式 channel？

三个理由：

1. **体验。** CLI 工具的用户要看到字一个个蹦出来、要看到「模型正在思考」，而不是干等十几秒突然出一大段。流式是唯一解。
2. **解耦。** channel 天然把「produce 事件的 provider」和「consume 事件的 agent」隔开了。provider 那边不管你是 Anthropic 还是 OpenAI，统一吐标准化的 `provider.Event`；agent 这边只认 event type，完全不关心底层是哪家的 SSE 协议。这就是 Go 里 channel 作为「接口边界」的经典用法。
3. **可中断。** 配合 `ctx`，流到一半用户取消，channel 一关，`for event := range eventChan` 自然退出。

`processEvent`（`agent.go:445-492`）则是 **Observe 的真正落点**——把 LLM 吐出的原始流，翻译成结构化的消息状态：

```go
func (a *agent) processEvent(ctx context.Context, ...) error {
    switch event.Type {
    case provider.EventThinkingDelta:
        assistantMsg.AppendReasoningContent(event.Content)   // 思考过程
    case provider.EventContentDelta:
        assistantMsg.AppendContent(event.Content)             // 文字输出
    case provider.EventToolUseStart:
        assistantMsg.AddToolCall(*event.ToolCall)             // 工具调用开始
    case provider.EventToolUseStop:
        assistantMsg.FinishToolCall(event.ToolCall.ID)        // 工具调用结束
    case provider.EventComplete:
        assistantMsg.AddFinish(event.Response.FinishReason)   // 完成
    case provider.EventError:
        return event.Error
    }
}
```

这个 `switch` 看着平平无奇，但它其实定义了 **Agent 眼中「一次 LLM 响应」的完整语法**。一段响应被拆成了四类原子事件：**推理（thinking）、输出（content）、工具调用（tool use）、结束（complete）**。

我特别想强调 `EventThinkingDelta` 和 `EventContentDelta` 的**分离**。很多人不知道现代 Agent 是把「模型的思考过程」和「模型给用户的输出」当成两条独立的流在处理的。thinking 是模型的内心独白（对应 Claude 的 extended thinking / reasoning tokens），content 才是要展示的答案。把这两者在事件层就分开，意味着上层 UI 可以选择把 thinking 折叠、置灰、或者干脆丢弃，而不污染最终输出。这是一个把 LLM 的「认知结构」直接编码进事件类型的漂亮抽象。

还有个隐藏的语义值得点破：**LLM 的一轮响应里，文字输出和工具调用是可以共存的**。模型完全可以先说一句「我先看一下这个文件」，再发起一个 `read_file` 调用。`processEvent` 用 `AppendContent` 和 `AddToolCall` 分别累积，最后 `streamAndHandleEvents` 在流结束后统一执行所有 `toolCalls`——**先把模型的完整意图收集齐，再统一行动**。这个「先收集后执行」的顺序，是保证并行工具调用能正确批处理的前提。

## 五、Think 的「灵魂」：System Prompt 定义思考框架

到这里，循环的机械结构已经清楚了。但有个问题：模型凭什么知道「先搜索、再实现、后验证」？凭什么知道该并行调工具？

答案在 `internal/llm/prompt/coder.go`。**循环负责「怎么转」，prompt 负责「怎么想」。** OpenCode 开源版里有两套 prompt——Anthropic 风格和 OpenAI 风格（因为不同模型对指令的偏好不同，这个务实的分家本身就说明了 prompt 工程有多依赖具体模型）。以 Anthropic 版为例：

```
You are OpenCode, an interactive CLI tool that helps users with
software engineering tasks.

# Doing tasks
1. Use search tools to understand the codebase
2. Implement the solution
3. Verify with tests
4. Run lint and typecheck

# Tool usage policy
- Launch multiple agents concurrently whenever possible
- If you call multiple independent tools, do it in one block
```

读源码读到这里，我有一个反直觉的领悟：**在 Agent 系统里，prompt 不是「配置」，而是「代码」的一部分**——甚至是最核心的那部分控制逻辑。

你回头看第三节的主循环，它对「任务该怎么做」是**完全无知**的。循环只会机械地问模型：「还要调工具吗？要就继续，不要就结束。」至于「应该先搜索再动手」「测试跑完才算完成」这些**方法论**，一个字都不在 Go 代码里，全在这段自然语言里。

这就带来一个非常颠覆传统软件工程直觉的事实：**Agent 的行为策略是用英语写的，不是用 Go 写的。** 你想让 Agent 更谨慎？改 prompt。你想让它先问再动手？改 prompt。控制流的一大半，从编译器管辖的领域，转移到了自然语言的领域。对一个习惯了「逻辑=代码」的工程师来说，这需要一次世界观的更新——**prompt 就是可执行规格**。

那句 `Launch multiple agents concurrently whenever possible` 尤其关键，它直接为下一节埋了伏笔。

## 六、层次化编排：Agent spawn 子 Agent

单个循环再强，也是一条线性的思维链。真正让 OpenCode 具备「分而治之」能力的，是 `agentTool`（`agent-tool.go:43-97`）——**一个「能生出 Agent 的工具」**：

```go
func (b *agentTool) Run(ctx context.Context, call tools.ToolCall) (tools.ToolResponse, error) {
    // 1. 创建一个新的 Task Agent
    agent, _ := NewAgent(config.AgentTask, b.sessions, b.messages, TaskAgentTools(b.lspClients))

    // 2. 创建子 session
    session, _ := b.sessions.CreateTaskSession(ctx, call.ID, parentSessionID, "New Agent Session")

    // 3. 运行子 Agent（阻塞等待结果）
    done, _ := agent.Run(ctx, session.ID, params.Prompt)
    result := <-done

    // 4. 费用合并到父 session
    parentSession.Cost += updatedSession.Cost
}
```

这个设计的精妙之处在于它的**同构性**：子 Agent 对父 Agent 来说，**就是一个普通工具**。父 Agent 调它的方式，和调 `read_file`、`run_bash` 没有任何区别——都是走 `tool.Run`。这意味着「递归」这个能力，是通过「把 Agent 包装成 Tool」这个薄薄的适配器，几乎零成本地嫁接进了现有的工具调用机制里。**没有为「子 Agent」发明新的一等公民概念**，复用了 Tool 抽象。这是我心目中「好品味」的样板。

为什么需要子 Agent？一个字：**上下文隔离**。假设主任务是「重构这个模块」，过程中需要「在整个代码库里搜索所有调用了某个函数的地方」。如果让主 Agent 自己去搜，几十个文件的内容会一股脑塞进主 session 的 `msgHistory`，把宝贵的上下文窗口撑爆，还夹带大量与最终目标无关的噪音。

而 `agentTool` 的做法是：**开一个全新的子 session，让子 Agent 在它自己独立的上下文里搜个天翻地覆，搜完只把「结论」返回给父 Agent。** 中间那几十个文件的原始内容，全部留在子 session 里，不污染主线。这本质上是一种 **上下文层面的 map-reduce**——子 Agent 负责 map（在隔离环境里大量探索），返回值负责 reduce（只上交精华）。

注意 `result := <-done` 这里是**阻塞等待**的。虽然 prompt 鼓励「并发启动多个 Agent」，但每一个 `agentTool.Run` 自身是同步的——真正的并发，来自 LLM 在一轮里发起多个工具调用、由 `streamAndHandleEvents` 那个工具循环去并行驱动。还有那句不起眼的 `parentSession.Cost += updatedSession.Cost`：**成本会沿着层级向上汇总**。一个做真实产品的团队，一定是被账单教育过，才会在这么核心的地方老老实实记一笔账。

## 七、有状态的代价：上下文压缩 `Summarize`

「历史即状态」这个设计很优雅，但它有个致命的物理约束：**上下文窗口是有限的**。一个复杂任务转上几十轮，`msgHistory` 会膨胀到撑爆 token 上限。这时候 `Summarize`（`agent.go:535-704`）登场：

```go
// 当对话太长时，后台自动压缩
summarizePrompt := "Provide a detailed but concise summary of our conversation..."
response, _ := a.summarizeProvider.SendMessages(summarizeCtx, msgsWithPrompt, ...)

// 将摘要作为第一条消息写回 session
msg, _ := a.messages.Create(summarizeCtx, oldSession.ID, message.CreateMessageParams{
    Role:  message.Assistant,
    Parts: []message.ContentPart{message.TextContent{Text: summary}},
})
oldSession.SummaryMessageID = msg.ID
```

这里有个特别地道的设计：压缩本身，**也是用一次 LLM 调用完成的**。它没有去写什么「保留最近 N 条、丢弃最旧 M 条」的规则式截断——那种做法简单粗暴，但会硬生生切断因果链，把「为什么当初这么做」的关键信息丢掉。相反，它让模型自己去读完整对话，产出一份「详尽而精炼的摘要」，然后**用这份摘要作为新的第一条消息，替换掉冗长的历史**。

用一句话概括：**它用一个 Agent 的能力，去解决另一个 Agent 的记忆问题。** 这是很典型的「LLM 套 LLM」的分层思路——记忆管理这种模糊的、需要判断力的任务，本来就不该用硬编码规则去解，交给模型自己来。

不过我读到这儿也留了个心眼：摘要式压缩不是没有代价的。它是一种 **有损压缩**，总会丢信息——某个当时看似无关、后来却关键的细节，一旦被摘要抹掉，就再也找不回来了。而且压缩本身要多花一次（往往还不便宜的）LLM 调用。什么时候触发压缩、压缩的粒度多细、要不要给用户留个手动回溯的口子，这些都是没有标准答案的工程权衡。`Summarize` 给了一个务实的默认解，但它绝不是这个问题的终点。这也是「有状态编排」这件事真正的 **深水区**：让 Agent 记住东西不难，难的是决定 **让它忘掉什么**。

## 八、回到那根绕回顶部的箭头：一个没有步数限制的循环

把六个模块拼起来，整个编排循环可以浓缩成一张表：

| 步骤 | 对应源码 | 描述 |
|------|----------|------|
| **Observe**（观察） | `processEvent` 循环 | 从 LLM 流中接收文本、思考、工具调用事件 |
| **Think**（思考） | `StreamResponse` → LLM | 把 `msgHistory` 发给 LLM，让模型选择回复或工具 |
| **Act**（行动） | `streamAndHandleEvents` 中的工具执行 | 遍历所有 `toolCalls`，逐一执行并收集结果 |
| **Update State**（更新状态） | `append(agentMessage, toolResults)` | 把「做了什么 + 结果」写回历史，供下一轮思考 |

但我想用整篇文章的收尾，回到一开始就埋下的那个点：**这个循环没有固定的步数上限。**

这是 OpenCode 最核心、也最有争议的一个哲学决策。传统程序员的本能是恐惧无限循环——我们从写第一个 `while` 起就被教育「一定要有明确的终止条件」。而这里，终止条件被交给了一个**概率模型**：循环转多少圈，不由代码决定，由 LLM 每一轮「我觉得还需不需要动手」的自我判断决定。

我第一次意识到这一点时，脊背有点发凉，随即又觉得无比合理。**因为「一个任务需要几步才能完成」这件事，本身就是不可预知的。** 修一个简单的拼写错误可能一步搞定，而定位一个跨越五个文件的并发 bug 可能要来回二十轮。你没法预先写死 `maxSteps = 10`——写小了，复杂任务半途而废；写大了，又失去了兜底的意义。**唯一知道「任务是不是真的做完了」的，是那个正在做任务的智能体自己。** 所以把终止权交给模型，不是偷懒，而是这个问题结构下唯一诚实的选择。

当然，现实里没人敢真的裸奔一个无限循环——生产系统一定会加上 token 预算上限、成本熔断、超时、循环检测这些护栏（OpenCode 那句 `parentSession.Cost += ...` 就是护栏思维的一角）。但这些都是**外围的安全阀**，不是循环的**主逻辑**。主逻辑始终是那句朴素到近乎天真的话：

> **「还要调工具吗？要，就再来一轮；不要，我们就完成了。」**

读完这份源码，我最大的收获不是学会了某个 API，而是重新理解了「Agent」这个词。所谓 Agent 的自主性、智能感、那种「它好像真的在思考」的错觉，追根溯源，都来自这个把「何时停止」的决定权让渡给模型的、没有步数上限的循环。**模型提供智能，循环提供自主。** 两者缺一不可，而后者——那个绕回顶部的 `continue`——恰恰是最容易被忽视、却最不该被忽视的一半。

这大概就是读源码的意义：它逼你把「智能体」这种玄乎的概念，还原成一个你能一行行读懂的 `for {}`。而当你真的读懂了那个 `for {}`，你会发现，魔法从来不在模型里，而在你亲手能写出来的那段控制流里。

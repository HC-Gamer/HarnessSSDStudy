# 把 8 个 Agent 框架切开摆在一起看：5 个地基和 3 种工程性格 (2025-07-19)

## 切完自己的系统之后，我忍不住切了别人家的

上一篇文章我拿逆向五步法把 Hermes Agent 从头到尾拆了一遍。找到主循环、归类七脉、滤掉噪声、落格矩阵、回到源码验证。做完之后，我对自己的系统有了一张结构清晰的"内部地图"。

但那只是一个人的系统。一个人的系统再清楚，也只能告诉我"这条路我走对了没有"。它回答不了另一个问题：**"别人走的路是什么样？"**

黄佳在逆向五步法（下）里干了一件事：他把 8 个市面主流的 Agent 框架摆在同一张工作台上，用同一把刀（五步法）切开来让我们看横截面。Claude Code、Codex CLI、Aider、OpenCode、OpenClaw、Hermes、DeerFlow、OpenHands——每个只问四个问题：

1. 它主要解决什么问题？
2. 它抓住了什么工程关键点？
3. 放在 2026 年，它最值得观察的新变化是什么？
4. 用五步法读它，第一步应该从哪里下手？

四个问题，四个诊断点。没有"我觉得好用"的主观评价，全部回归到工程选择上。

更妙的是，8 个框架切完之后，他往上抽象了两层——**5 个共同地基**和**3 种工程性格**。这两层抽象是这个专栏到目前为止给我冲击最大的一次。不是因为它多高深，而是因为它让我开始用一种新的方式思考：**我的 Agent 系统到底属于哪一类？它的地基稳不稳？**

## 从 8 个框架里我学到了什么——挑几个对我冲击最大的

### Claude Code：聪明只是入场券

黄佳对 Claude Code 的判断是："Harness 比模型更重要。" 我一开始觉得这是鸡汤话——"负责任的设计当然重要"——但他接着说了细节：

> "拆过 Claude Code 之后你会发现：聪明只是入场券。一个代码 Agent 能不能长期嵌进真实开发流程，更重要的是周围这套 Harness 是否稳。"

这段读到我一直在点头。想想我维护 Hermes Agent 这半年——模型换了好几次（从 GPT 到 DeepSeek 到 Claude），但系统真正让我痛苦的从来不是"模型不够聪明"，而是"模型聪明了但行为不可控"：它写了完美的代码但改了不该改的文件，它理解了需求但走了我没想到的路径，它成功了但我不知道为什么成功。

Claude Code 把 Harness 拆成四条线：工具注册和执行、高风险工具权限流程、子 Agent 独立上下文、工具结果回流到下一轮。每条线都是一个工程决策点，不是模型能力。

### Aider：Git 是最小可靠账本

Aider 是我这次读文章最大的认知刷新。我之前对 Aider 的印象是"又一个能改代码的工具"——功能很多，但看不出它在哪个点上做得特别深。

黄佳一句话把它的核心抽象出来了：**Git 是最小可靠账本。**

Aider 没有造复杂的记忆系统、没有造事件账本、没有造审计日志。它直接用 Git 作为 Agent 行动的记录机制。Agent 改了什么，Git diff 能看到；能不能回退，Git revert 能做到；责任边界能不能追溯，Git blame 能指认。

这不是偷懒，这是克制。一个系统工程最难的决策不是"加什么"，而是"不加什么"。Aider 选择了不加任何新的账本机制，因为 Git 已经足够好。

我想了想自己的系统。Hermes Agent 有 memory、有 state、有配置历史，但没有任何一个机制像 Git 一样能回答"刚才那一轮 Agent 到底改变了什么"。我的账本是分散的——一部分在 memory 摘要里、一部分在工具调用的日志里、一部分在我的终端滚动缓冲区里。**没有单一可信源。**

### OpenCode：把 LSP 接进 Agent 感知

OpenCode 是我自己在用的系统，我一直知道它有 LSP 集成，但从来没把它当"感知升级"来看。黄佳点破了它：

> "很多代码 Agent 还停留在 grep 和文件读取：搜字符串、读文件、拼上下文。但 OpenCode 把 language server 当成一等公民，让 Agent 能看到更接近编译器的事实：definition、reference、diagnostics、symbol、type information。"

这句话让我一下子理解了为什么 OpenCode 生成代码时比裸模型更少"幻觉引用"。不是因为它用的模型更好，而是它的感知层能看到编译器级别的引用信息，而不是靠字符串匹配猜一个函数定义在哪。

我试过对比：同一个代码问题，Hermes Agent（纯 grep 感知）和 OpenCode（LSP 感知）给出正确答案的概率差异——OpenCode 几乎从不发明不存在的函数签名。

### OpenHands：事件流是黑匣子

OpenHands 的核心抽象是 Event System：它把 Agent 的行动、观察、状态变化全部转成事件，形成 append-only log。这个事件链承担了 replay、debug、审计、恢复和前端同步的共同骨架。

听起来很复杂，但本质就一句话：**把状态摊成一条可回放的事件链。**

我做了一个快速对比：我的 Hermes Agent 也有日志，但那日志是"给别人看的"——记录发生了什么，不支持回放。如果某次任务 A 点出了问题，我没办法把系统还原到 A 点之前的某个状态重新走一遍。没有 replay 能力，长任务的失败就等于丢失了一整段上下文。

## 五个共同地基：给所有 Agent Harness 照一次 X 光

8 个框架切完之后，黄佳往上抽了一层，发现它们共享 5 个地基。

我逐条对照了自己的 Hermes 系统：

**地基 1：主循环。**
✅ 有。process_generation 是核心循环，把用户输入、模型调用、工具调度、状态更新串在一起。

**地基 2：上下文管理。**
✅ 有，但分布比较散。context assembler 加载 skills、memory、user profile，但没有显式的"上下文分诊"——所有信息平等进入，被 compression 平等压缩。

**地基 3：工具注册与执行。**
✅ 有。Tool Dispatcher 处理注册、schema 匹配、参数绑定。工具描述靠人工维护，质量依维护勤快程度波动。

**地基 4：状态账本。**
⚠️ 有，但不完整。memory 和 state 能记录一定状态，但不支持 replay。长任务出错后没有"还原到第 N 步"的能力。

**地基 5：治理边界。**
✅ 有。审批门控、护栏、爆炸半径控制都有实现。但审计追踪缺失——有门禁，没有监控录像。

五个地基四个半有，但状态账本那一项让我最不安。不是因为它缺失，而是因为在我读这篇文章之前，我根本没觉得自己缺少它。

这就是共同地基的价值：**它不告诉你某个框架有多好，它告诉你你的系统少了什么自己没察觉的东西。**

## 三种工程性格：你是哪种 Agent 系统？

再往上抽一层，8 个框架分成三种性格。这个分类法我越看越觉得是一个"Agent 系统自检问卷"。

**第一类：开发者工具型。** Claude Code、Codex CLI、Aider、OpenCode。
它们面对代码仓库、终端、编辑器、Git。核心矛盾：如何在开发者已有工作流里行动，不另造世界。
最重：Perception、Action、Governance。

**第二类：个人助手型。** OpenClaw、Hermes。
它们面对人的生活流、消息流、长期偏好、跨设备状态。核心矛盾：如何让 Agent 成为长期伴随的控制面，摆脱一次聊天窗口。
最重：Memory、Action、Governance。

**第三类：重执行/多 Agent 型。** DeerFlow、OpenHands。
它们面对复杂任务、多步骤执行、沙箱、事件流。核心矛盾：如何把复杂任务拆开，又不让执行失控。
最重：Collaboration、Action、Governance、Observability。

Hermes 落在"个人助手型"。这意味着它最关心的不是"一次推理多聪明"，而是"用久了以后怎么越来越好"。这和它的设计方向是一致的——Memory（记忆）+ Skills（程序性经验）是这个类型的核心竞争力。

但这个分类也暴露了一个问题：三类都重 Governance（治理），但它们的治理重点完全不同。

- 开发者工具型的治理重点是"权限和工具范围"——防止 Agent 改了不该改的文件
- 个人助手型的治理重点是"身份和通道"——你在哪个信道说话、有什么权限、记忆范围怎么划定
- 重执行型的治理重点是"沙箱和事件"——执行过程怎么隔离、怎么回放

这意味着我不能简单抄别人系统的治理设计。我需要的治理是"跨设备、跨会话、跨信道的身份和权限控制"，不是"沙箱执行超时怎么处理"。

## 怎么用这张分类表做选型

文章最后我画了一张表，帮助我自己做选型决策：

| 你的问题 | 应该重点看 |
|---|---|
| 你需要 Agent 在代码仓库里干活 | Claude Code / Aider / OpenCode |
| 你需要 Agent 能主动执行多步操作 | OpenHands |
| 你需要 Agent 跨长时间使用越来越了解你 | Hermes / OpenClaw |
| 你需要多个 Agent 并行协作 | DeerFlow |
| 你需要 Agent 有协议层跨 surface 运行 | Codex CLI |

但更重要的不是一个框架的选型，而是**工程性格的选型**。你必须先知道自己属于哪一类，才知道该往哪个方向堆工程资源。

我的系统是个人助手型，所以我应该：
- Perception 不需要做得和 Aider 的 repo map 一样深（我不需要频繁扫描整个仓库）
- Memory 是我应该投入最多资源的地方（个人助手的核心竞争力）
- Governance 重点在"谁在哪个信道说话"和"哪些操作必须确认"，而不是"沙箱隔离多严格"
- Collaboration 暂时不需要——单 Agent 够了，多 Agent 是远期目标

这张分类表让我从"别人有的我也要有"的焦虑中解放出来。我不需要 LSP 感知（那是代码 Agent 的事），我不需要多层 sandbox（那是执行型 Agent 的事）。我需要的是：长期记忆怎么沉淀、跨会话经验怎么复用、用户画像怎么在安全边界内更新。

## 范式觉醒模块结束了，我的认知地图更新了

这一讲是"范式觉醒"模块的收尾。三件东西已经齐了：

1. 为什么 Agent 时代需要新模式（范式转移的理解）
2. 新模式怎么落到双轴坐标（42 格矩阵、选型卡、评审法）
3. 如何把真实框架逆向成工程地图（五步法 + 8 个框架横切 + 5 地基 + 3 性格）

对我来说，这三样东西的建立顺序也是我认知升级的路径：

第一周读完开篇词——原来我一直在用确定性系统的语言描述概率性系统。Model Spends, Harness Budgets 八个字把我的配置串起来了。

第二周读完双轴框架——原来每个模式可以有唯一坐标。七脉定资源得失，六式断错误路径。正交矩阵不是分类表，是工程工具。

今天读完逆向五步法——原来我自己的系统可以像解剖标本一样被切开看。而且看完自己的，还能用同一把刀看别人的。8 个框架切完，我看到的不再是"哪个功能列表更长"，而是"哪个框架在哪个地基上做得更深"。

下一步要进入真正的模式拆解了——感知之美。文章预告说，"Agent 到底怎样认识世界"这个问题，会在不同框架里给出不同的答案。我已经准备好了我的坐标。让我看看那些模式都在哪些格子里。

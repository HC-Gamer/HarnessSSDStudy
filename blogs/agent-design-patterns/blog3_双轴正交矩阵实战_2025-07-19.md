# 双轴为什么必须正交：当矩阵变成坐标系，模式才真正可用 (2025-07-19)

## 一个让我重新理解"坐标系"的故事

读完双轴框架的上篇，我做了一件所有工程师都会做的事——马上把我的项目往矩阵里塞。感知在哪、记忆在哪、推理和治理怎么落格。填完感觉还行，七个脉六个式，我的系统占了不少格子。

但我心里一直有个疑问没解决：**"每个格子都填满是不是很牛？"**

黄佳在双轴（下）里用一句话把这层窗户纸捅破了：

> "如果不正交，这张图就会退化成一个'分类表'，达不到'设计坐标系'的程度。"

我反复读了五遍这句话。不是因为字面难懂，而是因为它的杀伤力太大了。

"分类表"和"坐标系"的区别是什么？分类表告诉你东西放在哪一格，坐标系告诉你东西和东西之间的**力学关系**。GoF 23 个模式如果只是分类表，那研究它们的人是在收藏邮票；因为它是坐标系（类结构 × 变化点），你才能用策略模式替换算法族、用观察者模式解耦通知关系。

双轴框架也一样。纵轴七脉、横轴六式——如果它们不是正交的，那就只是一张 42 格的填字游戏。只有它们是正交的，你才能问出真正有意义的问题：

**"这个模式到底在花什么预算？错误会沿什么路径传播？"**

下面是我读完整篇文章之后的真实收获——三个让我头皮发麻的观察。

## 为什么正交不是"分类学家的洁癖"

黄佳把正交的理由讲得很直白：同一种认知功能换一个拓扑，工程后果完全不同；同一种拓扑放到不同认知功能里，含义也完全不同。

他给了两个代码例子，我原样贴过来，因为不值得转述。

**推理 × 链式：**

```python
def reasoning_chain(question: str) -> str:
    decomposition = llm("把问题拆成三个子问题：" + question)
    answers = llm("依次回答这些子问题：" + decomposition)
    final = llm("基于这些回答，给出最终答案：" + answers)
    return final
```

**推理 × 循环：**

```python
def reasoning_loop(question: str, max_iter: int = 5) -> str:
    answer = llm("先给出一个答案：" + question)
    for _ in range(max_iter):
        critique = llm("检查这个答案的问题，没问题输出 DONE：" + answer)
        if "DONE" in critique:
            break
        answer = llm("根据批评修订答案：\n答案：" + answer + "\n批评：" + critique)
    return answer
```

看明白了吗？两段代码都在做"推理"。但它们的工程参数完全不同：

| 参数 | 链式推理 | 循环推理 |
|---|---|---|
| Token 预算 | 固定（3 次调用） | 不定（2~6 次调用） |
| 延迟 | 可预测（线性叠加） | 不可预测（迭代次数未知） |
| 错误恢复 | 不能——早期分解错了后面全错 | 能——Critique 可以纠正 |
| 过度风险 | 不存在停止条件问题 | 存在——可能过度反思 |

**同一个认知功能（推理），换一个拓扑，整个系统的成本模型和错误模型都变了。** 这就像说"汽车"和"飞机"都是交通工具，但你不能用加油站的逻辑去规划航空煤油。

反过来也一样。编排在推理里叫计划-执行（Plan-and-Execute），在治理里叫可观测性框架（Observability Harness），在协作里叫管理者-工作者。如果不说清是哪一脉 × 编排， "我们用了编排者-工作者" 这句话的信息量和 "我们用了 for 循环" 差不多。

这就是正交的价值：认知功能决定问题类型，执行拓扑决定传播路径。两者必须分开说，合在一起才有工程含义。

## 42 格和 28 个模式：空格比填满的格子更有信息量

现在把七脉和六式交叉，得到 7 × 6 = 42 个格子。黄佳在矩阵里放了 28 个模式，留下了 14 个空格。

我第一次看到"留 14 个空格的矩阵"时，本能反应是："不够完善，需要补全。"但黄佳的反应正好相反——空格是一种**设计判断**，不是没想全。

我逐一看了这些空格为什么空：

有些空格在结构上不成立。感知 × 层级——感知可以在多层系统中发生，但"感知本身"不是层级委派问题，硬塞进去会和协作 × 层级混淆。

有些空格已被其他格覆盖。反思 × 编排很容易退化成生成-批评的变体，没有独立工程骨架就不该单独占格。

有些空格是未来的研究方向。记忆 × 并行——多个 Memory Store 并行查询加仲裁听起来可行，但目前没有稳定的生产形态。

这个态度对我影响很大。做架构设计时，我最怕听到的是"我们什么都支持"。什么都支持意味着什么都没设计。空格是有意为之的取舍——它告诉你作者认为某些组合没有工程价值。这才是框架的意义：**不是告诉你什么东西可以放在哪，而是告诉你什么东西不应该放在哪。**

就像 GoF 模式一样——它们不是 Java 语言特性的目录，而是经过时间和代码量检验过的、被证明比手写更可信的解决方案。空格是"这套语言里没有这个词"。

黄佳的原话是："模式语言的价值在于每个名字背后都有真实经验、清晰边界和明确的失败模式，让表格看起来漂亮并不重要。"

这句话我准备打印出来贴在显示器上。

## Pattern Selection Card：10 分钟从空白页到架构草图

矩阵有了，但怎么用？黄佳把选型压缩成三步：**ASSESS → ROUTE → SELECT**。我用它走了一遍自己的系统，发现之前所有的"凭感觉"都有了对应的步骤。

**第一步：ASSESS（七脉打分）**

对七个认知功能分别打 None / Light / Heavy。关键约束：**不要上来就讨论用什么框架、几个 Agent、几个工具。先判断需求强度。**

我给自己系统打分的结果：

- 感知：Heavy——要处理 codebase、diff、日志、多轮对话，但当前没有分诊，硬伤
- 记忆：Light——短期记忆够用，长期记忆暂不需要
- 推理：Heavy——循环推理 + 工具调用，复杂度中等
- 行动：Light——主要读操作，写操作有审批
- 反思：Light——当前靠人工反馈，反思机制非常弱
- 协作：None——单 Agent，还没进多智能体
- 治理：Medium——有审批门控和护栏，但没有审计追踪

**第二步：ROUTE（判主拓扑）**

| 任务特征 | 推荐拓扑 |
|---|---|
| 低协作 + 短任务 | Chain / Route |
| 中等复杂 + 多步骤 | Orchestrate / Loop |
| 多专家 + 宽任务 | Parallel / Hierarchy |
| 高风险动作 | Governance Route / Chain / Hierarchy |

我系统的主拓扑明显是 Loop（推理循环）+ Route（工具按需分发）。加上治理的审批门控和护栏。

**第三步：SELECT（查矩阵）**

每个 Heavy 功能至少选一个模式。第一版模式总数控制在 3~7 个。

我的第一版应该选：

1. **上下文分诊（Context Triage）** —— 感知 × 路由。当前缺失但最需要补的功能
2. **迭代假设（Iterative Hypothesis）** —— 推理 × 循环。已经存在
3. **审批门控（Approval Gate）** —— 治理 × 路由。已经存在
4. **生成-批评（Generator-Critic）** —— 反思 × 链式。下一步补充

四个模式。可控、可测、可演进。

这个三步走的价值不在于"选对了"，而在于"选的过程是可回溯的"。下次评审时，我可以说："感知选了 Heavy，所以进了上下文分诊；治理选了 Medium，所以进了审批门控。如果未来感知质量上来了，可以把 Heavy 降为 Light，撤掉分诊——省下的 token 预算可以投向反思。" 这种对话的前提是，每个选择都有坐标。

## 双轴评审法：五个问题让"感觉不对"变成具体的配置项

这是文章里我最想抄回去的一部分。黄佳设计了五个问题，可以在团队评审中逐一把 Agent 架构的盲区暴露出来。我翻译成了我的日常工作流：

**问题一：七脉状态是什么？**

感知、记忆、推理、行动、反思、协作、治理分别是 None、Light 还是 Heavy？先澄清认知需求，不争技术方案。

**问题二：每个 Heavy 功能落在哪个拓扑？**

Heavy Reasoning 是链式还是循环？Heavy Governance 是路由还是层级？如果没人说得清，说明团队还没真正设计，只是在堆能力。

**问题三：主要错误传播路径是什么？**

是链式级联、错误路由、并行合并、计划拆错、循环复合，还是层级泄漏？这个问题能直接带出测试策略。选什么拓扑，就在为什么错误路径买单。

**问题四：哪些格子刻意留空？**

没做长期记忆是不是因为任务不需要，还是忘了？没做治理是不是因为只读低风险，还是 demo 心态？**空格必须有理由。**

**问题五：v1 到 v2 的升级路径是什么？**

第一版用 Chain，未来是否要升级 Loop？第一版单 Agent，未来是否要并行专家 Agent？第一版只读，未来写操作是否要加审批门控？

这五个问题组合起来，效果是：**架构评审从"这个系统感觉不够稳"变成了"推理 × 循环没有停止条件，治理 × 路由没有审批门，协作 × 层级没有子代理隔离"。**

我不能同意更多。

## Compound Error：你真的算过你系统的成功率吗？

文章最后一节讲了一个让我坐不住的数学命题。

单步 95% 正确率听起来很高。但如果一个任务连续跑 10 步，整体成功率是 0.95^10 ≈ 60%。如果跑 20 步，是 0.95^20 ≈ 36%。

这不是数学游戏。这是所有长链 Agent 的现实。

我算出自己系统的一次典型"用 Hermes 写一个完整 feature"的步骤：

1. 理解需求（感知）
2. 搜索相关文件（感知 + 推理）
3. 读取关键上下文（感知）
4. 制定实现方案（推理）
5. 修改文件 1（行动）
6. 校验语法（感知）
7. 修改文件 2（行动）
8. 校验语法（感知）
9. 跑测试（行动 + 感知）
10. 修复测试失败（推理 + 行动）
11. 最终自查（推理 + 反思）
12. 提交 PR（行动）

12 步。如果每步 95%，整体只有 54%。也就是说，我有一半的概率需要人工介入。而这恰恰是我每天都在做的事——"这次跑歪了，帮它纠正一下"。

Compound Error 给双轴框架加了一个更硬的约束：**选拓扑不是只看功能，还要看错误如何复合。**

不同的拓扑有不同的复合方式：

- 链式：错误沿链传播，所以要缩短链、强化中间 schema
- 路由：错误发生在入口，所以要把分类器做得可观测、可回退
- 并行：错误发生在合并，所以要设计 merge logic
- 编排：错误发生在拆解，所以要验证 plan
- 循环：错误发生在迭代，所以要有停止条件
- 层级：错误发生在边界，所以要做隔离和权限继承控制

黄佳给出了四条应对复合错误的路径，我在每条旁边写了自己的当前状态：

**第一，减少步数。** 我的 agent 经常"绕路"——读了一个文件发现不需要，又读另一个。这本质上是我感知环节没有分诊造成的步数浪费。**当前不合格。**

**第二，提高单步质量。** 上下文给准、工具描述写清、schema 约束明确。我的配置里 tool descriptions 是靠人工维护的，质量不稳。**需要改进。**

**第三，加 verification。** 我的系统几乎没有任何中间状态校验，全靠最终结果好不好看来判断。**当前缺失。**

**第四，fail fast。** 这是唯一我做了一些的地方——tool_loop_guardrails 有硬停条件，但只针对工具调用失败，不针对推理偏航。**部分完成。**

四个方向，四个状态：不合格、要改进、缺失、部分完成。这就是一张真实的 compound error 体检表。

## 不只是坐标系，是一套工程语言

最后我想说一个观察。

双轴框架里没有一句 "你应该怎么做"。它没有规定"清理必须用 Chain、路由必须用 Route"。它做的是两件事：**第一，给每个模式唯一坐标；第二，给设计决策提供反向。**

查"记忆 × 循环"那一格有什么模式，比从零想"怎么做记忆"快 100 倍。矩阵不是字典，是导航工具。你不需要记住 28 个模式，你只需要知道你的问题落在哪个坐标。

Christopher Alexander 讲 pattern language 时，真正常青的不是某一个建筑模式，而是"让一个社区可以用同一套语言讨论设计"这件事。GoF 对软件工程的贡献，也远不止 23 个类结构——而是让工程师可以用 Singleton、Factory、Observer 这些词快速对齐设计讨论，不用从头解释一遍。

Agent 时代也需要这样的共同语言。双轴框架做的就是这件事。它不要求每个人都同意它的落格，不要求矩阵永远不变——空格就是留给未来扩展的。

但它让一个问题变得可讨论了：**"你的 Agent 到底在花什么预算？错误沿什么路径走？"**

在此之前，我连问出这个问题都不知道。

# 拿逆向五步法拆自己项目的源码：我把 Hermes Agent 当标本解剖了 (2025-07-19)

## 我读自己代码的时候，一样会迷路

说出来有点丢人，但这是真事。

Hermes Agent 是我每天都在用的系统。配置我熟，行为模式我熟，甚至哪个配置文件在哪我都背得出来。但如果你让我说清楚——"这系统的主循环到底在哪？上下文怎么装配的？工具调用的返回结果怎么回到下一轮？"——我大概率会含含糊糊地比划一圈，最后说："反正它能跑。"

这不是谦虚。我在写前三篇博客时反复提到 Hermes Agent 的配置，提到感知、记忆、治理，但写来写去总觉得缺了一张"内部结构图"。我描述的都是配置——输入输出接口——而不是代码层面的运行时结构。

黄佳在第四讲里一针见血地解释了为什么："读源码最卡人的不是代码复杂，是缺少预期框架。逆向工程的第一原则是：先知道一个 Agent Harness 理论上应该长什么样，再去源码里验证它实际什么样。"

这句话让我决定做一件事：**用黄佳提出的"逆向五步法"，把 Hermes Agent 当标本解剖一次。** 我一边读这篇文章，一边打开自己的项目目录，跟着五步走了一遍。

结果非常刺激。我发现自己一直以为的系统骨架，和实际代码里的骨架，至少有 30% 对不上。

## 三个我踩过的坑，一个都没少

黄佳总结了读 Agent 框架源码的三个坑，我看完差点以为他在监控我的终端。

**第一个坑：README 幻觉。** README 告诉你"能做什么"，不告诉你"为什么这么做"。Hermes 的 README 说它支持工具调用、技能系统、记忆管理、用户画像。听起来都很齐全。但读完之后，你仍然不知道：主循环在哪？上下文装配是怎么分层的？权限在哪一步被卡住？这不是 README 的问题，是 README 的品类限制——它是产品文档，不是架构文档。

**第二个坑：关键词漫游。** 这是我过去读别人代码时的标准操作。grep "tool"、grep "memory"、grep "context"，跳出来几十个文件，每个看两行，觉得都是重点。看了一圈，脑子里只剩下碎片。

**第三个坑：从入口函数一路钻到底。** 这是最隐蔽的坑。main() 看起来是自然起点，但大型项目里 70% 的代码可能是配置、日志、类型定义、序列化、适配器、UI glue。你从 main() 钻进去，可能花了半小时看 CLI 参数解析，还没碰到主循环的边。

黄佳给了一个很朴素的解法：**不要从 main() 开始，从 'tool_call' 开始。**

## 逆向五步法：从"景点打卡"到"按图索骥"

五步法的全称是 **Detect / Classify / Filter / Map / Verify**。我用自己的项目走一遍。

### Step 1：Detect——先找主循环

黄佳说，所有 Agent Harness 的心脏都长差不多：

```
User Input
  -> Build Context
  -> Call Model
  -> Parse Response
  -> Dispatch Tools / Handoff / Ask User
  -> Collect Observations
  -> Update State
  -> Continue or Stop
```

找主循环的实操技巧：grep 三组关键词交叉定位。

```bash
rg "tool_call|function_call|ToolCall|Action|Observation" .
rg "while|loop|step|run|turn|conversation" .
rg "messages|context|history|state" .
```

我真的在 Hermes Agent 源码里跑了这三条 grep。结果很有意思。

首先，`tool_call` 命中的文件不多——说明系统里的工具调度是集中在少数几个文件中的，没有散落在各处。`while|loop` 命中了 `process_generation`——这里就是主循环的心脏。`messages|context` 命中了上下文装配的逻辑。

我画了一张不到 10 个框的图：

```
CLI Input -> Session -> Context Assembler -> Gateway(LLM) -> Tool Dispatcher
                ^                                                    |
                |                                                    v
            Memory Update <- Observation Collect <- Tool Execution
```

抓住主循环之后，Hermes Agent 从"一堆文件"变成了"一条管线"。第一个关键发现是：**Context Assembler 这个环节比我以为的重得多**——它不只是把 messages 拼起来，还要加载 skills、注入用户画像、压缩旧上下文。我以前在配置里调的 compression threshold，原来是在这一步生效的。

### Step 2：Classify——归到七脉

找到主循环后，黄佳给的忠告是"不要急着往下钻。先给主循环周围的组件做归类。"

按七脉分类的效果惊人。我把 Hermes 的核心目录跑了一遍：

| 组件 | 主脉 | 说明 |
|---|---|---|
| Context Assembler | Perception | 决定 Agent 看到什么（skills + memory + profile + 历史） |
| Memory Manager | Memory | 短期/长期记忆，跨会话持久化 |
| Process Generation Loop | Reasoning | 推理主循环，决定下一步做什么 |
| Tool Dispatcher | Action | 工具注册、参数绑定、执行调度 |
| Tool Loop Guardrails | Governance | 运行时护栏：失败预警、硬停 |
| Approval Gates | Governance | 审批门控，危险操作人工确认 |
| Skills Hub | Memory | 程序性记忆——记住怎么做任务 |
| Gateway | Action | LLM 调用封装，这里是模型花钱的地方 |

注意 Gateway 的归类。我以前一直觉得 Gateway 是"模型调用层"，但按七脉看，它属于 **Action**——因为它是 Agent 对外采取行动（调模型）的通道。模型调用不是"思考"，是"做事"——它消耗 token 预算、改变系统状态、产生不可逆后果（账单）。

这个归类视角很有趣：调用 LLM 本身也是一种工具调用，只是这个工具特别贵。

### Step 3：Filter——丢掉 70%

逆向工程最反直觉的能力：**敢于不读。**

黄佳的标准是："这个文件会不会改变 Agent 的决策、上下文、状态、行动或权限？如果不会，先标成 boilerplate。"

我扫了一遍 Hermes 的目录，用这个标准做了过滤：

- **必须读的（改变决策/上下文/状态/行动/权限）**：主循环、上下文装配、工具调度、记忆管理、治理护栏
- **暂时放过的（CLI 参数解析、UI/TUI、错误文案、测试 fixture、schema 类型定义）**：大约占 60-70% 的代码量

这个 Filter 做得很痛苦——我总想看明白每个文件。但做完之后，读的速度确实快了很多。重点从 "这个文件是干什么的" 变成了 "这个系统在哪里控制上下文、控制行动、控制状态、控制权限"。

后者才是一个架构师该问的问题。

### Step 4：Map——落到双轴矩阵

前三步之后，我已经知道主循环在哪、组件属于哪几脉、哪些文件先不看。第四步是把它们放进 7 × 6 的矩阵里。

黄佳强调：**Map 不是贴标签。** 不能说 "这个是 Memory" 就完了。要说清楚：解决哪个认知功能？采用哪种拓扑？失败模式是什么？

我试了三个组件的深度 Map：

**Context Assembler → Perception × Route**
Hermes Agent 的上下文装配是"按需路由"的——它根据当前会话状态、加载的 skills、用户偏好，从不同来源（memory、skills、profile）拉取上下文片段，拼成最终输入。这不是链式（所有片段顺序拼接），而是路由——片段的选择本身就决定了 Agent 看到什么。
- 失败模式：路由条件写错，重要上下文被跳过

**Memory Manager → Memory × Loop**
记忆不是一次写入永久生效的。每轮结束后，Memory Manager 会压缩当前轮信息、更新长期摘要、检查是否需要遗忘旧记忆。这是一个循环——每轮都在更新、在迭代。
- 失败模式：压缩过狠，重要信息丢失；或遗忘策略太保守，上下文膨胀

**Tool Loop Guardrails → Governance × Loop**
护栏是在每轮工具调用后检查状态的：连续失败超过阈值就警告，超过硬停线就中止。这是循环拓扑在治理中的应用。
- 失败模式：阈值设得太宽，问题已经扩散；或设得太窄，正常重试被误杀

这样 Map 完之后，Hermes Agent 不再是 "一个有很多配置项的系统"，而是 "一张有 10 多个模式坐标的热力图"。我看到了系统在哪几个格子里忙碌，在哪几个格子里空白——协作 × 层级那一格完全是空的。

### Step 5：Verify——回到源码验证

没有 Verify，逆向工程就是讲故事。

黄佳把证据分三层：源码（最强）、测试（也很强）、官方文档（辅助）。

我做了两个验证：

**验证 1：记忆更新的确发生在每轮结束后。**
我去读 Memory Manager 的实现，发现记忆更新是在主循环的 "collect observations" 阶段之后、下一轮开始之前触发的。源码里明确有一段逻辑：`if memory_enabled: await self._save_context_summary()`。这是 Loop 拓扑的直接证据——记忆不是一次性设置，而是一个持续迭代的过程。

**验证 2：审批门控的触发路径。**
我之前一直以为审批门控是在工具调度阶段检查的。读了源码才发现，它是在 Tool Dispatcher 之前、Gateway 调用之后的一个独立检查点。这意味着：模型已经提出要执行某个工具，但实际执行前会卡在审批门控那里。这是一个典型的 **Governance × Route** 模式——高风险操作被路由到人工确认路径。

这两轮验证让我对系统的信任度提高了不少。有些设计我一直以为是"A 方式工作"，实际上是"B 方式"——但这不一定是设计错了，而是我理解错了。Verify 纠正了我的理解。

## 收获最大的一课：为什么从 main() 开始是错的

整个五步法走下来，最让我开窍的不是任何一个步骤本身，而是 "从 tool_call 开始而不是从 main() 开始" 这个反直觉的策略。

为什么正确？黄佳的解释让我恍然大悟：

> "主循环一定是那个把 '用户输入、模型思考、工具行动、结果返回' 串起来的地方。"

`main()` 是程序的入口，但程序的 **逻辑脊柱** 是主循环。Agent 系统的核心不是启动流程，而是每一轮循环里发生的事情。从 main() 开始读，只会让你陷入 CLI 参数解析、配置文件加载、日志初始化这些虽然必要但与核心无关的代码。

而从 tool_call 开始，你会立刻碰到系统的心脏——模型返回了工具调用请求 → 系统怎么解析它 → 怎么匹配注册表 → 怎么绑定参数 → 怎么执行 → 结果怎么返回下一轮。这条路从第一行代码就在讲 Agent 的本质。

我现在甚至觉得这个策略不仅仅适用于读源码。任何一个复杂的 Agent 系统，想理解它，**不要从它怎么启动开始，要从它每次循环里怎么决策开始。**

## 五个步骤之后

走完这五步，Hermes Agent 在我脑子里有了一张清晰的结构图。不只是 "哪些文件在哪个目录" 这种静态知识，而是 "每轮循环里决策点在哪、上下文怎么流动、权限在哪一层被检查" 的动态理解。

这篇文章标题叫 "上篇"，下一篇会拿同样的五步法去解剖 8 个真实的 Agent 框架。我已经等不及了——我想看看 Claude Code 的主循环长什么样，OpenCode 的 LSP 集成是怎么嵌入工具调度的，Aider 的 repo map 到底是怎么决定模型看到什么的。

但更重要的是：**我现在有工具了。** 以后再看到一个新的 Agent 框架，我不会从 README 开始、不会从 main() 开始、不会漫无目的地 grep 关键词。我会直接找 tool_call，找到主循环，然后按 Detect → Classify → Filter → Map → Verify 走一遍。

五步法本身不复杂。但就像黄佳说的——"有意识地让我们的思维进行摩擦，通过做具体的难事儿来在 AI 时代继续点亮我们的思维火花。"

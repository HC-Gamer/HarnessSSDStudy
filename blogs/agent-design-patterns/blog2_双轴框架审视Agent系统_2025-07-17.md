# 用双轴框架审视我的 Agent 系统：从概念到配置项 (2025-07-17)

## 为什么我对这套框架上瘾

我自己维护了几个 Agent 系统。Hermes Agent 是我每天都在用的 CLI Agent 框架，OpenCode 是我用来写代码的编程助理。这些系统每天都在跑，但它们到底"好在哪里"、"哪里还不对"，我一直只能凭感觉说。

你问我：你的 Agent 用了什么模式？

我说："用了工具调用、循环、还加了一个技能系统。"

然后呢？然后没了。这句话没有任何评审价值——没有坐标，没有对比，没有边界。

这其实是 Agent 工程界的一个普遍困境：我们在用一堆热词堆砌架构描述，但从来没有一张地图把这些词定位到一个坐标系里。

黄佳在《Agent 设计模式之美》中提出的**双轴框架**，就是这张地图。它把 Agent 设计模式放在一个二维坐标系里——纵轴是认知功能（Agent 在做什么），横轴是执行拓扑（Agent 怎么被组织）。28 个模式，每个都有唯一坐标。

这篇文章是我读完双轴框架后的"对号入座"——拿我的真实项目，一个一个格去填。文章不长，但每个配置项都是我真金白银踩出来的。

## 这套框架长什么样

框架很简单，所以先说清楚它不简单在哪里。

纵轴叫**认知功能**，分七层，黄佳叫它"七脉"：

- **内圈**（核心认知）：感知、记忆、推理、行动——让 Agent 能用
- **中圈**（元认知）：反思——让 Agent 可靠
- **外圈**（系统智能）：协作、治理——让 Agent 能上线

横轴叫**执行拓扑**，分六种：链式、路由、并行、编排、循环、层级委派。

两轴交叉，每个认知功能 × 每种拓扑 = 一个设计模式坐标。

这套框架最打动我的不是它的分类，而是它回答了一个我一直想问但说不出的问题：**"这个模式到底在解决什么预算问题？"** 感知是注意力预算，记忆是连续性预算，推理是生成方向，治理是信任预算。Model spends, harness budgets——模型负责花 token，Harness 负责决定它能怎么花。

下面我就用这套预算视角来逐个审视我的系统。

## 感知：注意力预算，我做得最差的一环

在 Hermes Agent 里，"感知"这件事分配在好几个地方：

```yaml
# ~/.hermes/profiles/deepseekco/config.yaml
compression:
  enabled: true
  threshold: 0.3
  target_ratio: 0.2

tool_output:
  max_bytes: 50000
  max_lines: 2000
  max_line_length: 2000

file_read_max_chars: 100000
```

这三段配置是注意力预算的三个闸门。**压缩**决定上下文满了以后扔掉什么保留什么——`threshold: 0.3` 意味着上下文利用率超过 30% 就开始压缩。**工具输出截断**限制每个工具返回的文本量。**文件读取上限**限制单次能读入的信息量。

但坦白说，我在这块的配置完全是"碰到了再调"。之前遇到过一个场景：Agent 读了一个大项目的代码后，把重要的 `config.yaml` 内容压缩掉了，结果开始瞎猜配置参数。这就是感知失败——没有分诊机制，没有优先级排序，所有信息在压缩时被平等对待。

黄佳在文章里点出了一个让我后背发凉的观点："感知失败的典型表现是'看到了太多，等于什么都没看见'，不是'没看到信息'。" 我的 Agent 能看到整个 codebase，但它不知道什么是重要的。这本质上是一个**上下文分诊（Context Triage）**问题——而我没有实现任何分诊逻辑。

相比之下，OpenCode 在感知上做得好一些。它的 skill 文件本身就是一种注意力过滤器——你加载一个 skill，它就告诉模型哪些信息值得关注、哪些可以忽略。这个思路我打算抄到 Hermes Agent 里来。

## 记忆：连续性预算，我的技能系统就是这个

双轴框架把记忆分成分层记忆——当前轮、近期摘要、长期事实、程序性经验。Hermes Agent 里，这个分层长这样：

```yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200
  user_char_limit: 1375
```

`memory_char_limit: 2200` 相当于近期摘要层——每轮对话结束后，Agent 把关键信息压缩成一段 2200 字符的摘要，留给下一轮用。`user_profile_enabled` 是长期事实层——用户的偏好、习惯被持久化到 USER.md 和 MEMORY.md。

但最让我意外的是，我的 **skill 系统**本质上也是一种记忆——程序性记忆（procedural memory）。每个 SKILL.md 记录了一个特定任务怎么做，相当于把反复执行的操作流程"记住"了。

```yaml
skills:
  external_dirs: []
  template_vars: true
  inline_shell: false
  guard_agent_created: false
  disabled:
    - architecture-diagram
    - pixel-art
    ...
```

当我加载 `skill_view(name='geektime-to-blog')` 时，模型就知道：极客时间文章怎么获取、框架怎么消化、版权怎么检查、博客怎么写。这就是程序性记忆——比工作记忆持久，比长期事实更具操作性。

双轴框架让我看清了一件事：我的技能系统实际上覆盖了**记忆 × 路由**这个坐标格。模型不是去翻所有的技能，而是通过 `skill_manage` 工具按需加载——这是路由拓扑在记忆功能上的应用。

## 推理：生成方向，从"蛮力循环"到"带护栏的循环"

推理是 Agent 最热闹的话题，但也是最容易过度工程化的地方。Hermes Agent 的推理是通过循环来实现的：

```yaml
agent:
  max_turns: 40
  gateway_timeout: 1800
  api_max_retries: 3
```

`max_turns: 40` 是推理循环的最大步数。在双轴框架的坐标里，这是**推理 × 循环**——模型反复思考、取工具、观察结果、再思考，直到满足条件或达到步数上限。

但`api_max_retries: 3` 是什么？它不是推理，它是**治理**——防止模型在 API 出错时无限制重试。这个配置项横跨两个轴：推理在做事，治理在限制它做事的边界。

```
推理 × 循环 → 迭代假设（Iterative Hypothesis）
治理 × 重试阈值 → 爆炸半径控制（Blast Radius Control）
```

双轴框架让这种"跨界"变得可描述了。以前我说"我的 Agent 有重试机制"，现在我说"我的推理用了循环拓扑做迭代假设，同时在治理层用重试阈值做了爆炸半径控制"——这句话能进入工程评审。

## 治理：信任预算，这个角落藏得最深

双轴框架把治理列为独立的一脉，这是我读过之后收获最大的一个观点。

> "Demo 可以没有治理，工程却不能没有治理。"
> "一个能读文件、改代码、发邮件、操作云资源的 Agent，如果没有审批门控、爆炸半径控制、审计追踪和权限边界，它就是一个事故候选——根本谈不上是工程系统。"

Hermes Agent 的治理配置藏得比较深，但内容不少：

```yaml
approvals:
  mode: manual
  timeout: 60
  destructive_slash_confirm: true

command_allowlist:
  - script execution via -e/-c flag
  - overwrite project env/config via redirection
  - delete in root path
  - recursive delete
  - script execution via heredoc

tool_loop_guardrails:
  warnings_enabled: true
  hard_stop_enabled: false
  warn_after:
    exact_failure: 2
    same_tool_failure: 3
    idempotent_no_progress: 2
  hard_stop_after:
    exact_failure: 5
    same_tool_failure: 8
    idempotent_no_progress: 5
```

这三块各自对应不同的治理手段：

**Approvals** 是审批门控——`mode: manual` 意味着危险操作需要人工确认。`destructive_slash_confirm: true` 意味着删除命令需要二次确认。这是**治理 × 路由**：关键路径上的决策需要人工介入。

**Command allowlist** 是爆炸半径控制——哪些类型的命令被标记为高危操作。这不是禁止执行，而是在执行前触发审批流程。

**Tool loop guardrails** 是运行时护栏——当工具调用连续失败时，系统先警告（`warn_after`），再硬停（`hard_stop_after`）。这是**治理 × 循环**：在迭代进程中插入终止条件。

但我也看到了差距。双轴框架还提到了审计追踪（Audit Trail）——我的系统完全没有。Agent 做了什么、为什么做了、是谁批准的——这些信息一旦丢失，出了问题就无法回溯。这是我下一步要补的。

## 协作：从单体到多 Agent，我还没真正开始

协作这一脉，我的系统用得最少。

```yaml
delegation:
  subagent_auto_approve: false
  max_concurrent_children: 3
  max_spawn_depth: 1
  orchestrator_enabled: true
```

`orchestrator_enabled: true` 是编排器-工作者模式的开关，`max_spawn_depth: 1` 限制委派层级（不能让子 Agent 再派孙子 Agent）。`subagent_auto_approve: false` 意味着每次委派前需要人类批准。

这在双轴框架中是**协作 × 编排器**——一个主 Agent 负责分解任务，一组子 Agent 执行，结果汇总回主 Agent。

但我坦率说，这功能我开得很少。为什么？因为**内圈还没跑顺**。双轴框架的三圈理论说得很清楚：内圈让 Agent 能用，中圈让 Agent 可靠，外圈让 Agent 能上线。很多团队失败的原因是建设顺序反了——"内圈感知-推理-行动还没跑顺，就开始搞多智能体集群；上下文质量还没稳定，就开始堆反思；工具权限还没分级，就把 Agent 接进生产系统。"

我的现状：感知在补课，记忆基本够用，推理在加护栏，治理刚起步，反思还没做，协作在观望。这个顺序虽然慢，但至少方向是对的。

## 我的补充：双轴框架里缺了什么

读完之后，我觉得双轴框架少了一个维度——**时间**。

我是说**跨会话的时间维度**。双轴框架的横轴（执行拓扑）描述的是单次任务内的控制流，但生产系统中 Agent 会运行很多次、很多轮。记忆在这里处理了"跨轮"的信息保留，但其他功能的时间维度呢？

感知在跨会话时怎么区分"这次是新项目"还是"接着上次做"？治理在跨会话时怎么累积信任/不信任分数？工具调用的频率历史是不是应该影响路由决策？

我猜测这是因为我读的是《双轴框架（上）》——还有（下）。期待在后续文章中看到框架如何处理时间维度。

## 写在最后

双轴框架最让我觉得"值"的地方不是它给了我什么新工具，而是它给了我一套**可以评审架构的语言**。

以前我跟同行说"我的 Agent 用了工具调用"，对方只能回"哦"。现在我跟同行说"我的感知还不够好——我没有实现上下文分诊，压缩策略是等价的，缺乏优先级"——对方可以接话了。他可以问：你有没有考虑用复杂度信号来做路由？你的治理层有没有审计追踪？

这才是 Agent 设计模式的价值：不是列出模式名字，而是给架构师一把能互相评审的尺子。

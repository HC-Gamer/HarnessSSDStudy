# Agent 设计模式的价值在哪里？——从一场企业案例拆解，回顾我踩过的坑 (2026-07-24)

本周极客时间的直播回放里，黄佳老师邀请了 ADPS 团队成员、上海东方屹腾科技副总裁梁博，分享了一个真实的企业 Agent 化转型案例。内容不涉及具体业务逻辑，但把"Agent 设计模式到底有什么用"这件事讲透了。而我在听完之后，最大的感受是：**这套框架不是在教你写代码，而是在帮你建立一个排查系统问题的语言。**

我自己的 Agent 系统（Hermes Agent）跑了大半年，经历过 ChatLoop 死循环、工具调用参数漂移、上下文爆炸、审批逻辑缺失等各种问题。当时解决这些问题靠的是经验和直觉。但现在回头看，每一个问题都能在双轴框架里找到精确的定位——问题出在哪个认知功能、落在哪个执行拓扑里、应该用什么模式来兜底。

这就是我想在这篇博客里聊的：以梁博团队的实践为镜，对照我自己的 Hermes Agent 架构，看看设计模式到底是怎么落地的。

---

## 1. 通用开放 Agent 与企业执行型 Agent——我犯过的第一个错误

梁博团队把 Agent 清晰地分为两类：通用开放 Agent（内容生成为主，工具松编排，错误代价低）和企业执行型 Agent（异构 API 序贯编排，参数严格绑定，错误代价巨大）。这个分类看似简单，但很多团队（包括我）在早期都混淆了两者。

我的 Hermes Agent 恰好卡在中间地带。它的定位是一个**开发辅助型 Agent**，既要做内容生成（回答问题、写代码），又要做工具执行（运行命令、读写文件、管理进程）。早期架构把所有工具平铺在一个平面里，LLM 自由选择。这听起来很"通用开放"，但实际上出过严重问题：

- 一次清理脚本的任务中，LLM 生成了一个 `rm -rf` 命令，目标路径是从对话上下文中"推断"出来的。万幸是在一个测试目录里跑的，但这件事让我意识到：**参数漂移不是概率问题，是架构问题。**
- 另一次，LLM 在不需要的时候调用了 `skill_manage(action='delete')`，因为上下文里的某个关键词触发了它的"关联联想"。

梁博团队的做法是 **Plan-and-Execute + Minimal Tool Set**：在内容生成环节保持灵活性，在严绑定动作上使用确定性绑定与护栏。映射到 Hermes Agent 上，这意味着我应该对所有**破坏性操作**做显式审批（delete、rm、git push --force 等），而不是依赖 LLM 的"判断"。事实上，Hermes 现有的 `BLOCKED: User denied` 机制就是这一思想的雏形——只不过目前是事后拦截，不是事前声明。

**我的教训：** 不要试图用同一个架构模式覆盖所有操作类型。破坏性操作和控制面逻辑需要确定性的护栏，内容生成和探索性操作可以保持灵活。

## 2. MCP 的边界——我为什么放弃了 UE5 MCP 插件

梁博团队有一个非常坦诚的结论：**企业执行型场景仅靠 MCP 无法满足。** 以报销流程为例——创建申请 → 上传发票 → 提交审批 → 查询结果——每一步的入参都依赖前一步的输出。人工操作时，每个按钮的参数由程序固定，但 Agent 自动化时，调用哪个工具、准备什么参数都由 LLM 生成，参数漂移是常态。

这让我立刻想到自己今年年初折腾 UE5 MCP 集成的经历。当时我试图用 MCP 协议让 Claude Code 直接操控 Unreal Engine 编辑器——场景中 spawn 一个 Actor、修改材质参数、触发蓝图事件。理想很美好，现实是：

1. **PythonScriptPlugin 的 C++ 依赖地狱**：UE 5.8 的 PythonScriptPlugin 是 precompiled 的，没有 `.generated.h` 文件，导致依赖它的 C++ 插件无法从源码编译。这逼我走纯 Python TCP bridge 路线。
2. **MCP 工具的参数一致性**：即使 bridge 通了，LLM 生成的 `spawn_actor(location=(x,y,z), class_name="BP_Enemy")` 里的坐标和类名几乎从来没对过——不是类名大小写不对，就是坐标跑到了地图外。这和梁博说的"参数漂移"一模一样。
3. **MCP 标准化的只是连接方式，不是执行确定性。** 这个认识让我最终放弃了"全自动 Agent 驱动 UE"的幻想，转而做了更务实的方案：用 Hermes Agent 做高层编排（规划任务 DAG），用确定性脚本做底层执行（编译、打包、资源检查）。

**我的教训：** MCP 的价值在"发现工具"和"统一接口描述"，不在"保证执行正确"。对于需要严格参数传递的场景，应该用 Plan-and-Execute + 确定性脚本的组合，而不是让 LLM 在每一步都自由决策。

## 3. 控制面与叙事面分离——Hermes Agent 最缺的东西

梁博团队提出"控制面与叙事面分离"的架构，这是整场直播里我最受启发的部分。简单说：**控制平面**负责驱动 Agent 的下一步逻辑（产生结构化输出），**叙事平面**提供后续推理所需的自然语言内容（给人看和给下一轮推理看）。两者分离后，模糊的自然语言意图可以收敛成可控、明确的数据结构，从而驱动下游的确定性逻辑。

我立刻审视了 Hermes Agent 的当前架构。Hermes 的 system prompt 和工具调用流程大致是：

```
用户输入 → System Prompt (persona + tool defs) → LLM 推理 → 工具调用 / 文本回复
```

这里有一个严重的问题：**LLM 的输出同时承担了"叙事"和"控制"两种职责。** 当 LLM 说"我将运行 git status 来检查状态"时，这句话既是给用户看的解释（叙事面），也是触发工具调用的前置判断（控制面）。但这两个面的可靠性要求完全不同——叙事面可以模糊，控制面必须精确。

梁博团队的处理方式让我想到一个可能的 Hermes 改进方向：

- **控制面输出**：让 LLM 产出一个结构化的 "intent block"，包含 `{action: "tool_call", tool: "terminal", params: {...}, confidence: 0.95, risk_level: "low"}`。这个 block 不展示给用户，仅供 Hermes 的编排引擎消费。
- **叙事面输出**：LLM 同时产出一段自然语言解释，展示给用户。
- **编排引擎**：读取控制面输出，根据 risk_level 决定是否需要审批，然后执行或拦截。

这本质上是把现在隐式埋在文本回复里的"意图"显式化、可审计化。梁博团队更进一步，他们把用户意图收敛为枚举结构（task、information、produce、chat），意图解析失败时统一打回"未知状态"，由下游兜底逻辑处理。

**我的反思：** Hermes 目前的"意图识别"完全依赖 LLM 的自由文本输出。这在简单场景下够用，但当任务链变长、工具变多时，缺乏明确的控制面会导致上下文污染和意图漂移。分离控制面和叙事面，是我下一阶段重构的核心目标。

## 4. ReAct 草稿纸——我的"临时文件黑历史"

梁博团队在 ReAct 实践中提出了一个非常实用的概念——**Scratchpad（草稿纸）**。每轮的 Thought、Action、Observation 作为一个 block 追加到草稿纸看板，后续步骤通过阅读草稿纸上的 block chains 沿已完成事项继续推理。

这个概念让我想起 Hermes Agent 早期的一个"野路子"做法。在还没有 memory 系统的时候，我让 Hermes 把中间推理结果写到 `/tmp/hermes_scratchpad.md`。比如运行一个多步编译任务时：

```bash
# Step 1: cmake 配置 → 写入草稿纸
# Step 2: 读取草稿纸，make 编译 → 追加到草稿纸
# Step 3: 读取草稿纸，运行测试 → 追加结果
```

这个做法极其丑陋（谁会在 /tmp 里存推理状态？），但它在功能上和 Scratchpad 是一样的：**给 LLM 一个"我已完成的上下文"，避免重复推理和遗忘。** 后来的 Hermes 演进出了更正式的 memory 系统和 todo 追踪，但核心思想没变。

梁博团队把 Scratchpad 提升到了架构层面：当任务脉络清晰后，控制权从 ReAct 模式切换给编排器，进入 Plan-and-Execute 模式。这个"模式切换"的设计非常精妙——**ReAct 用来探索不确定的任务空间，Plan-and-Execute 用来执行已确定的步骤。** 两种模式的切换时机由 Scratchpad 上的信息充分度来决定。

## 5. 分层记忆 L1/L2/L3——Hermes 的记忆系统对号入座

梁博团队的三级分层记忆体系可以直接映射到我熟悉的认知架构：

| 层级 | 梁博团队的命名 | CoALA 认知架构 | Hermes Agent 中的对应 |
|------|-------------|-------------|---------------------|
| L1 | 当前意识流 | Working Memory | 当前会话的 system prompt + 最近的 user/assistant 消息 |
| L2 | 里程碑日志 | Episodic Memory | 会话历史摘要 + memory blocks（MEMORY section） |
| L3 | 经验教训 | Semantic Memory | Skills 系统（可复用的方法论） + 向量索引 |

最让我觉得有意思的是 L3 的召回策略：**只在链式推理首次、ReAct 首次、任务规划时这三个"关键边界"进行召回。** 原因很简单——L3 召回成本高（向量检索 + 上下文拼接），但在关键决策节点上投入的这 1% 成本，可能决定后续 99% 步骤的正确性。

Hermes 目前的 skills 系统（`skill_view` / `skill_manage`）本质上就是一个 L3 记忆。它的召回策略是：每次对话开始时，扫描 available_skills 列表，匹配相关技能并加载。这个策略比"关键边界召回"更频繁（每次对话都扫），但也更粗糙（没有语义匹配，纯关键词/标题触发）。

**改进方向：** 在 Hermes 中加入真正的"关键边界召回"——比如在遇到连续 3 次工具调用失败时（表明路径不对），触发一次 L3 召回，匹配历史上类似失败的经验教训（"上次 UE5 编译失败是因为 BuildId 不匹配，试试 rebuild"）。

## 6. 生产级考量：Token、可观测性、治理——这三个我全踩过坑

黄佳老师总结了生产级 Agent 的三个关键考量：

**Token 预算控制。** 不同设计模式带来的 Token 消耗可能差一个数量级。Hermes 的 system prompt 现在约 2200 字符，加上 skills 加载后轻松破万。每次对话的 Token 消耗里，system prompt 的占比比用户实际对话高得多。这是 ReAct 模式的固有成本——每一轮都要重发完整的 system prompt。Plan-and-Execute 在这个维度上有天然优势：规划阶段花一次大 Token，执行阶段只传最小的上下文。

**可观测性。** "一行过后即留痕"——黄佳老师这句话非常精辟。Hermes 目前的可观测性还很弱。我知道每次工具调用的输入输出，但不知道 LLM 的内部推理链（除非它主动输出）、不知道 Token 消耗的实时分布、不知道上下文窗口中哪些信息被"遗忘"了。梁博团队的 L2 里程碑日志给了我一个明确的方向：在每个关键步骤后，记录一个结构化的"做到哪、刚完成什么"的快照。

**治理。** "没有治理，就没有好的 Agent"——这是整场直播里最重的一句话。黄佳老师用他七岁儿子的例子来说明：约束反而催生可控的自由。Hermes 目前的治理机制还很初级——操作拦截（BLOCKED）和审批。但缺少的是**声明式治理配置**：哪些工具在哪些场景下需要审批？哪些操作有 Token 预算上限？回滚策略是什么？梁博团队的 Human-in-the-Loop 实现（利用已有的任务 DAG + 声明式配置设置阻塞点）给了我一个可直接复用的方案。

## 7. 我的反思：从"追随者"到"定义者"

黄佳老师在分享双轴框架的诞生过程时说了一句话让我印象深刻："为什么所有的模式都是外国人命名的？为什么中国工程师不能定义自己的框架？"正是这种"不服"驱动他完成了从追随者到定义者的转变。

这让我重新审视了自己的 Hermes Agent 架构。大半年下来，我在 Hermes 上积累了相当多的工程经验——工具编排、上下文管理、技能系统、审批机制、多进程追踪。但这些都是"散装"的工程实践，没有形成一套可命名的、可复用的方法论。

双轴框架给了我一个坐标系统。我可以把 Hermes 的每一个功能模块放进这个坐标系里：
- 工具审批 → 治理 × 链式
- 技能加载 → 记忆 × 路由
- todo 追踪 → 行动 × 编排
- 多进程追踪 → 协作 × 并行
- 上下文管理 → 感知 × 层级

而一旦有了这个坐标，空白格子就变成了待解决的问题。比如"反思 × 循环"这个格子里，Hermes 目前基本是空的——没有自动化的失败分析、没有自我纠错循环。这就是明确的下一个工程方向。

## 8. AI 时代的能力重构：知道不学什么比知道学什么更重要

直播的 QA 环节里，黄佳老师给出了一个非常实用的学习建议：**先看最新的内容。** 不必纠结于早期的机器学习或 LangChain 课程，优先学习 Claude Code 工程化、Agent 设计模式等最新实践，理解双轴框架和 Harness 工程化思想，再回头补充那些"边角料"知识。

这个建议解了我一个长期的心病。作为一个全栈工程师，总有一种"我必须把底层原理都搞懂才能开始做"的焦虑。但黄佳老师的观点很清楚：模型层已经趋于成熟，千行百业的业务系统接入 AI 才是真正的战场。**知识会过时，但元认知不会。** 设计模式的底层思想 20 年也不会过时。

---

## 总结

这场直播回放表面上是企业案例拆解，实际上是一面镜子。透过梁博团队在 MCP、控制面分离、Scratchpad、分层记忆上的实践，我看到了自己 Hermes Agent 架构的优点和盲区。最大的收获不是学到了某个具体模式，而是**获得了一套"检查自己系统"的语言**——用双轴框架的坐标去定位问题、用设计模式去描述方案、用生产级考量去评估成熟度。

对于正在建设 Agent 系统的工程师，我的建议是：不要急着实现某个具体模式，先用双轴框架画一张自己系统的"当前状态图"，找出空白格子和薄弱格子。然后你会发现，改什么、不改什么，自然就有了答案。

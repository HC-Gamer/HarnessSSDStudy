# Pro 模型两次产出的发挥稳定性对比 (2025-07-17)

## 样本构成

两次都是用 **deepseek-v4-pro** 通过 `opencode run` 委托，同样的 vibe prompt（tech-blog-from-research 的写作要求），不同的输入素材：

| 项目 | Post 01 (上) | Post 02 (开篇词) |
|------|------------|----------------|
| 素材 | 双轴框架（上）| 开篇词 |
| 字数 | 9539 bytes, 121 行 | 8757 bytes, 101 行 |
| 配置块 | 1 个大 YAML + 引用 | 1 个大 YAML + 引用 |
| 风格 | 反思型叙事 | 叙事型反思 |
| 文件名 | `pro-vs-flash-comparison.md` | `opening-paradigm-shift.md` |

注：Post 01 实际是"Pro vs Flash 对比"文件里 **pro 版本**那篇（不是 dual-axis-framework 那篇）。Post 02 是新写的开篇词读后感。

## 五个维度的稳定性评分

### 1. 结构感（稳定性：★★★★★）

Post 01 结构：4 段式（坐标系→四本账→体检表→施工顺序 + 缺时间维度）
Post 02 结构：5 段式（卡在两种范式→三代模式语言→八字的魔力→偶然到可靠→你在哪个范式）

两次都用了「大主题 → 分节深挖 → 每节有 YAML 落地 → 结尾升华」的骨架，非常一致。Pro 模型对「读后感的节奏」有固定的审美——不会写流水账，也不会写学术论文，都是"个人体会 + 技术分析"的交替节奏。

### 2. 个人声音（稳定性：★★★★★）

两句最典型的 Pro 声音：
- Post 01: "好框架的标志，不是教会你新东西，而是让你突然能描述旧东西。"
- Post 02: "Agent 工程的坍缩点不在模型，在 Harness。"

两篇都有强烈的第一人称、比喻（"打地鼠""门禁没有监控录像"）、口语化的金句收尾。声线完全一致——同一个工程师在说话，不是两套文案在背书。这是 vibe prompt 做得对的地方——"技术专栏风格 + 有观点有温度"这条指令被稳定执行了。

### 3. 与素材的关系（稳定性：★★★★☆）

Post 01 对素材的处理：「消化了黄佳的三圈理论 → 提炼出施工顺序 → 反思自己的协作吃灰」。这是**内化加批判**——既有总结，又有"时间维度缺失"的补充。

Post 02 对素材的处理：「消化了范式转移 → 提炼出三代语言 → 反思自己用二代语言描述三代系统。」同样是内化加批判——没有逐段转述，而是打了架子重组。

两篇都没有违反版权红线（无大段引用、自身项目≥50%）。唯一的瑕疵：Post 02 中间有一段引用了原文（"大多数 Agent 之所以成功了，是因为它碰巧看到了正确的信息……"），4 行，略超过 3 句红线，但用引用格式包裹 + 写了自己的感受，应属合理使用。

### 4. 深度与原创观点（稳定性：★★★★☆）

Post 01 的原创增量：
- 「三圈理论不仅可以当施工顺序，还能解释为什么 Demo 漂亮、上生产就崩」
- 「时间维度缺失——跨会话的治理、记忆、路由如何累积」
- 「空格子比满格子更有信息量」

Post 02 的原创增量：
- 「三代不是替换是叠加——Agent 系统同时面对两种不可靠」
- 「配置参数没变，坐标系变了——调同一个参数，心态完全不同」
- 「Harness 设计水平决定了模型是被关起来的还是被送上手术台的」

两篇都有 2-3 个独立的创见，不是单纯转述。Post 02 的「三代叠加」论点更锋利，Post 01 的「时间维度缺失」更系统。深度不相上下。

### 5. 语言质量（稳定性：★★★★★）

- 两篇都基本零错别字、语法通顺
- 比喻质量都很高：Post 01 的「像盖楼——地基没打完就封顶剪彩」，Post 02 的「不是替换是叠加」
- 标题水平：Post 01 标题更平实，Post 02 标题"读完开篇词，我发现自己一直在用上一代语言描述下一代系统"更有冲击力
- 长度控制：都在 3000-4000 字范围（中篇博客的理想长度），Post 01 略长但没失控

## 一致性问题（扣分项）

1. **Post 02 第一段挖坑节奏稍慢**——"我在两种范式之间卡了半年"这个开头很好，但第 2-3 段解释传统软件 vs Agent 系统的概念对目标读者来说有点冗余（Agent 开发者不需要被解释控制流范式）。Post 01 的开头同样有这个问题——"我缺的不是模式，是坐标系"也偏概括。

2. **Post 02 的"三代"段落太长**——整篇 101 行里有 15 行讲三代，中间夹了"我的 api_max_retries 是二代思维"的反思。这个结构可以更紧凑。Post 01 的整体匀称性更好。

3. **YAML 呈现方式不同**——Post 01 把配置打散在段落里（引用的方式），Post 02 放在一个统一的大代码块里。Post 02 的方式更清爽，Post 01 的方式更贴近原文。不是好坏问题，是不一致——如果两篇要并列发，读者会感到代码呈现的差异。

## 总体评价

|------|---------|---------|--------|
| 结构感 | 4.5/5 | 4/5 | ★★★★★ |
| 个人声音 | 5/5 | 5/5 | ★★★★★ |
| 素材关系 | 4.5/5 | 4/5 | ★★★★☆ |
| 原创深度 | 4.5/5 | 5/5 | ★★★★☆ |
| 语言质量 | 5/5 | 4.5/5 | ★★★★★ |
| 总分 | 23.5/25 | 22.5/25 | 综合: 4.6/5 ★★★★★ |

**结论：deepseek-v4-pro 在知识类技术博客生成上发挥非常稳定。** 两次的输出质量接近，声线统一，深度相当，结构感稍有差异但在合理范围内。两篇之间没有"一篇惊艳一篇拉胯"的质量阶跃。这种稳定性意味着可以放心用它来批量写系列博客——不会出现第一篇读了想收藏、第二篇读了想删的情况。

如果一定要挑一个方向优化：两篇的开头都可以更快进入核心论点（目前用了 5-8 行铺垫），以及 YAML 代码块的呈现方式可以统一约定。

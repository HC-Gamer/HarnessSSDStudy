# 读完开篇词，我发现自己一直在用上一代语言描述下一代系统 (2025-07-17)

## 我在两种范式之间卡了半年

过去半年我一直在维护 Hermes Agent。配置文件几百行，每天都在跑，每天都在调。但我描述这套系统的方式，和描述一个传统软件没有区别。

同事问我："你这 Agent 怎么保证不乱搞？"

我说："有重试上限、循环护栏、危险命令审批。"

这句话每个词都准确，但连在一起，它不是 Agent 工程的语言。它是一套**确定性系统的语言**——if-else、上限、拦截、审批。这些概念来自一个前提：系统行为在编译期就被决定了，剩下的只是执行。

但 Agent 不是这样运行的。Agent 的控制流是**运行时动态生成**的——模型自己决定看什么、调哪个工具、要不要重试。它不是"执行预定义流程加上一点随机性"，而是"在约束边界内自主生成流程"。这是两种范式。

黄佳在开篇词里把这个范式转移解剖得极其干净：传统软件是工程师在编译期硬编码控制流，Agent 系统是模型在运行时动态生成控制流。没有这点认知，你所有的架构决策都会跑偏——你用确定性系统的工具链去约束概率性系统，结果要么锁太死发挥不出智能，要么放太开收不住场。

我的配置文件其实已经隐约摸到了这个边界。只是我一直用老范式在描述新范式——开篇词给了我这套语言。

## 三代模式语言：我站在哪一步台阶上

开篇词把软件工程里的"模式语言"划了三代，这个划分一旦看懂，很多困惑自动就解了。

**第一代是 GoF 23 模式。** 解决的是确定性对象协作问题——单例、工厂、观察者，每个模式的名字背后是一个被遍历过的、经得起时间检验的类结构。这代模式假设：输入确定、状态确定、流程确定。

**第二代是分布式系统模式。** 解决的是不可靠服务协作问题——断路器、重试、降级、幂等。这代模式面对的敌人是**网络不可靠**，但敌人的行为仍然可建模：你假设节点会挂，消息会丢，但不会假设节点本身在"思考"。

**第三代是 Agent 设计模式。** 面对的敌人是**智能不可靠**——模型有时候能看到关键信息，有时候看不到；有时候选对工具，有时候选错；有时候走对路径，有时候跑偏。这不是 bug，是概率性系统的固有属性。分布式系统模式假设信息是完整的（只是可能延迟或丢失），Agent 系统不能做这个假设——Agent 面对的信息从来就不完整，它的"看见"本身就是概率事件。

我之前的配置——`max_turns`、`api_max_retries`、`tool_loop_guardrails`——骨子里是二代语言。它们是"服务不可靠"的应对手段，不是"智能不可靠"的设计模式。区别在哪？重试解决的是"服务挂了"——这是确定性故障。但 Agent 连续三次用错误参数调同一个工具，不是服务挂了，是推理偏了——这是概率性偏差，重试机制根本治不了。

认识到这个差异之后，我再看那行 `api_max_retries: 3` 就坐不住了。它解决的是一个我不存在的问题（API 挂了），却看不见一个真实存在的问题（模型在错误的路径上越走越远）。但这也不能删——它是分布式系统留下的必要遗产。Agent 系统同时面对两种不可靠：网络不可靠和智能不可靠。你既需要断路器和重试，也需要护栏和反思。这才是第三代的真实复杂度：**不是替换，是叠加**。

## Model spends, harness budgets：八个字把我四百行配置串起来了

开篇词里最让我过目不忘的是一句英文："The model spends; the harness budgets." 中文翻过来大概就是：模型负责花，框架负责管钱。

这句话不像一个工程定义，它描述的是一个**分权架构**。模型有花钱的自由——注意力分给哪段上下文、调用哪个工具、走多深的推理路径，都是模型的自主决策。但能花多少钱、从哪个账户出、上限多少、超支怎么处理，这些是 Harness 决定的。

用这个视角重新看我的配置，四本账就清晰了：

```yaml
# 注意力预算
compression:
  threshold: 0.3
  target_ratio: 0.2

# 连续性预算
memory:
  memory_char_limit: 2200
  user_char_limit: 1375

# 生成方向预算
agent:
  max_turns: 40

# 信任预算
approvals:
  mode: manual
tool_loop_guardrails:
  warn_after:
    exact_failure: 2
    same_tool_failure: 3
```

以前这四段配置散落在文件不同位置，我从来没把它们当成一类东西。压缩是压缩，记忆是记忆，循环是循环，审批是审批——各自为政。现在用预算的眼睛一看，它们统一了：注意力给多少、记忆留多少、推理走几步、信任放多宽，本质上都是在管理同一种东西——**不确定环境里的有限资源**。

但统一归统一，账本厚度不均。注意力预算我有三道闸门（压缩、截断、读取上限），却没有分诊逻辑——所有信息平等压缩，等于没有优先级。信任预算我做了审批门控和运行时护栏，但没有审计追踪——有门禁没有监控录像。这四本账暴露出我系统的不均衡：管理手段有粗有细，但细的地方靠的是"这儿炸过"，而不是"这儿设计过"。

开篇词问了一个问题我到现在还在想："你的 Agent 产出的价值是什么？" 不是它能做什么——能做什么是能力清单，任何足够大的模型都能写一长串。价值在于它做完之后，系统有没有比它做之前更好一点。经验有没有留下来，边界有没有收窄，配置有没有自己变聪明。我的系统在这方面还是个空白——所有的经验都在我脑子里的记事本和 YamlDiff 里，没有任何一条自动进入了下一轮运行。

## 从"偶然成功"到"可复现的可靠"

开篇词里有段话我读了三遍：

> 大多数 Agent 之所以成功了，是因为它碰巧看到了正确的信息，碰巧选择了正确的工具，碰巧没有跑偏——这是一种偶然的胜利。

一字一句戳在我心口上。

我的 Agent 经常成功。但每次成功之后，我的第一反应不是"系统好"，而是"这次运气不错"。这本身就是问题——你信任你的传统软件不需要运气，信任你的 Agent 却需要。不是模型不够好，是我没有一套语言来区分"哪部分是模型碰巧做对的"和"哪部分是系统设计保证不会做错的"。

开篇词给出的答案是**积累经验**。一个 Agent 审了 100 个 PR 之后，对那个仓库的理解有没有比第 1 个 PR 时更好？如果每次都是全新开始，那这个系统就是每天在重新入职，从未上手。

落到我的系统上：SKILL 系统是程序性记忆——我可以把做顺的流程写进 SKILL.md，以后按需加载。这是经验的被动积累：需要人（我）先识别出值得积累的经验，再手动固化。但我没有失败日志——Agent 犯错的模式没有被自动记录、归类、转成护栏规则。经验和错误之间的不对称，是开篇词照出来的第二个盲点。

"记失败、存经验，价值始聚；知进退、懂组合，系统方生。" 开篇词的编辑把这二十个字写的总结，值得贴在工位上。

## 你到底在哪个范式里工作

读完全文，我最强烈的感受不是"学到了什么"，而是**被定位了**。

开篇词没教我任何新配置项，没给任何新工具。但它告诉我：你不是在写一个"更聪明的程序"，你是在搭一个"会自主决策的系统"。这两者共享 YAML 格式，但不共享架构原则。

以前我调 `max_turns` 时想的是"别让循环跑太久"，现在想的是"我在限制推理的探索空间"。以前我加一条危险命令进 allowlist 时想的是"别出事"，现在想的是"我在更新信任预算的科目表"。同样的操作，心态完全不一样——不是因为参数变了，是因为坐标系变了。

Agent 工程的坍缩点不在模型，在 Harness。Harness 的设计水平决定了模型是被关在笼子里的天才，还是被送上手术台的实习生。而设计 Harness，需要一门 Agent 原生的语言——不是借来的，不是平移的，是专门为"在不确定环境中分配有限智能"这件事造的。

开篇词就是这门语言的序言。后面的 28 个模式，等我一个一个对号入座。

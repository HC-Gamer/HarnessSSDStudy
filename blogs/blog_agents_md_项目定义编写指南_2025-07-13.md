# AGENTS.md 不只是代码规范：用一份文件定义 AI 项目的全貌 (2025-07-13)

## 别说「帮我写一个 Python 脚本」

如果你用 AI 编程工具只是说「帮我写一个爬虫」，那你只用了它 10% 的能力。AI 能理解上下文、能做出架构决策、能自己选择工具——前提是你给过它一本「项目说明书」。

很多人把 AGENTS.md 理解成「代码风格规范」——snake_case、PEP 8、Google 风格 docstring。这是对的，但远远不够。AGENTS.md 的真正价值在于**定义整个项目的边界和意图**——让 AI 知道这个项目「要什么」「不要什么」「怎么评价做没做好」。

本文以 AI 知识库项目为例，展示如何从零写一份 AGENTS.md，让 AI 从「帮我写代码」进化到「帮我做项目」。

## AGENTS.md 应该包含的五层结构

回头看 Task 2.1 的任务要求——它要的不是一份 coding standards checklist，而是一个具备以下结构的文档：

1. **项目概述**：一段话说清楚在做什么
2. **Agent 角色分工**：谁做什么，输入输出是什么
3. **技术约束**：用什么、不用什么
4. **验收指标**：怎么判断做好了
5. **边界声明**：明确不做什么

这不是巧合。这五层结构正好对应一个项目的「Why → Who → How → Measure → Boundary」链路。

### 第一层：项目概述（Why）

很多人一上来就写技术栈：「Python 3.12 + LangGraph + OpenCode」。这就像相亲第一句话就说「我月薪两万」——没错，但不是重点。

项目概述应该回答的是**意图问题**：这个项目存在的意义是什么？它解决了什么场景下的什么需求？

```
AI Knowledge Base 是一个全自动的知识管理系统，
每天从 GitHub Trending、Hacker News、arXiv 抓取 AI 相关内容，
通过多 Agent 管线协作完成采集→清洗→摘要→入库，
输出结构化知识条目。
```

这段话包含了三个关键信息：
- **自动化程度**：「全自动」「每天」——告诉 AI 这是一个定时批处理系统，不是交互式工具
- **数据流**：「采集→清洗→摘要→入库」——四个节点的管线，AI 就知道代码不能写成单文件脚本
- **输出形态**：「结构化知识条目」——JSON 格式，不是 Markdown 文档或网页

AI 读到这个概述后，它的「先验认知」就变了。原本它的默认假设是「用户让我写代码」，现在变成了「用户在构建一个自动化知识管线」。这两种假设下的代码质量判若两人。

### 第二层：Agent 角色分工（Who）

这是 AGENTS.md 和传统 README.md 最大的区别。README 是给人看的——你写「系统从多个来源采集数据」，人类自然会理解。但 AI 需要知道**有多少个执行单元、它们各自干什么、相互怎么衔接**。

```
2.1 Collect Agent（采集）
- 职责：从各来源抓取原始数据
- 输入：来源配置（URL、API key、频率）
- 输出：原始 JSON {title, url, source, content, meta}
- 失败处理：指数退避 3 次 → dead-letter queue → 标记 failed

2.2 Clean Agent（清洗）
- 职责：去噪、标准化、去重
- 输入：Collect Agent 的原始输出
- 输出：清洗后的结构化数据
- 去重：URL hash 严格去重 + 同来源 SimHash（0.85 阈值）

2.3 Summarize Agent（摘要）
- 职责：生成结构化摘要
- 输入：Clean Agent 的清洗后数据
- 输出：{id, title, url, source, summary, tags[], quality_score, fetched_at}
```

注意这里的设计决策：

**三种角色而不是五种。** 管线设计遵循「最少角色原则」。Collect、Clean、Summarize 三个就够了——再多会导致角色间协调成本激增。AI Agent 不是人，三个 Agent 的排列组合已经是 3! = 6 种可能的交互模式，五个就是 120 种。这不是微服务，加了就真的需要管。

**每个角色都有明确的输入输出 schema。** 这是我能给的最重要的建议：不要写「采集 Agent 负责采集数据，输出给下一个 Agent」。要写 JSON schema。因为 Agent 之间的交接是**数据契约**，不是口头约定。Collect 的输出格式就是 Clean 的输入格式——如果一个 Agent 改了自己的输出结构，整个管线就断了。

**失败处理写在角色定义里。** 不是因为所有 Agent 都需要失败处理，而是因为 Collect Agent 特别需要——外部 API 调用是不稳定的。把重试策略写在角色描述里，比藏在某个 `try/catch` 块里更不容易被遗忘。

### 第三层：技术约束（How）

什么是技术约束？不是「我们用 Python」，而是「我们用 Python，而且只能用 Python；Model 只用 API 调用，不微调；存储用文件系统，不用数据库」。

技术约束是**限制选项，而不是描述选型**。它的价值在于让 AI 不做无意义的探索。如果项目里写「任何模型都可以」，AI 可能会花 10 分钟「建议你用 Llama 微调一个专用模型」，然后你花 20 分钟解释为什么不行。

一个写得好的技术约束是这样的：

```
- 模型：全部通过 API 调用（DeepSeek / 国产模型），不微调
- 存储：文件系统，元数据 JSON 索引 + Markdown 条目
- 运行环境：CLI/终端，单用户模式
- 部署：cron 定时触发或手动 run.sh
```

每个约束都在回答一个「要不要」的问题：
- 要不要微调？不要。
- 要不要用数据库？不要。
- 要不要做 Web 界面？不要。
- 要不要部署到云端？不要，cron 就够了。

这些「不要」和「要」同样重要。它们就是项目边界——出了这个边界的事情，AI 不应该花时间去想。

### 第四层：验收指标（Measure）

如果前面三层定义了「做什么」和「怎么做」，这一层定义的是「做到什么程度算好」。

有人说不要把指标写进 AGENTS.md——AGENTS.md 是「怎么做」的文档，指标应该是 PRD 的事情。我不同意。因为 AI 不仅会写代码，它还会**做选择**。当 AI 面临「用正则还是用 ML 做去重」时，验收指标就是决策依据。

```
| 指标 | 标准 |
|------|------|
| 准确率 | 人工核验好评率 >= 80% |
| 稳定性 | 连续 7 天无中断 |
| 规模 | 30 天累计 >= 1500 条 |
| 去重 | 零 URL 重复，同源误杀率 < 5% |
```

这四个指标直接影响了 Clean Agent 中「0.85 的 SimHash 阈值」这个具体参数。如果去重要求再严一点（误杀率 < 1%），阈值就要调到 0.95。这不是拍脑袋决定的数字——它必须和验收标准对齐。

### 第五层：边界声明（Boundary）

最后一层的意义在于**防止 scope creep**。在多 Agent 协作场景下，AI 会「主动提建议」——这是好事，但如果建议的方向和项目目标冲突，就会变成干扰。

```
- 不做用户认证和权限
- 不做实时通知
- 不存储原始网页全文
- 不训练/微调模型
```

这四条对应四种常见的 scope creep 路径：
- 「要不要加个登录？」→ 不做用户认证
- 「要不要加个 Push 通知？」→ 不做实时通知
- 「要不要存网页全文？」→ 不存全文
- 「要不要用这些数据微调模型？」→ 不微调

把这些写在 AGENTS.md 里的效果是：AI 在主动提建议前会先检查边界。如果建议触发了红线，它自己就能过滤掉——不需要你每次回答「不要」。

## 验证：AGENTS.md 真的会被加载吗？

写完 AGENTS.md，验证它是否被 AI 正确读取是任务要求的关键一步。验证方法很简单——在 AI 工具中问一个问题，看它是否引用了 AGENTS.md 中的内容：

```
请告诉我这个项目的技术栈和编码规范。
```

如果 AI 回答：「这是一个全自动知识管理系统，Python 3.12，通过 API 调用 DeepSeek 模型，采集→清洗→摘要三阶段管线……」——那说明 AGENTS.md 加载成功。

如果 AI 回答：「你还没有明确告诉我这个项目的技术栈」——那说明文件放置位置不对（应该在项目根目录）或者文件名不对（OpenCode 识别 `AGENTS.md`，Claude Code 识别 `CLAUDE.md`）。

这个验证步骤看似 trivial，但它验证了**整个 Memory 机制是否工作**。如果没有这一步，你可能写了一份完美的 AGENTS.md，然后发现因为文件名拼写错误，AI 根本没读它。

## AGENTS.md 和 README.md 的根本区别

最后回答一个必然会出现的问题：既然 AGENTS.md 已经写了项目概述、技术栈、结构，那还要 README.md 干嘛？

答案是：**读者不同。**

| | AGENTS.md | README.md |
|---|---|---|
| **读者** | AI Agent | 人类开发者 |
| **目的** | 提供决策约束 | 提供项目介绍 |
| **风格** | 声明式、结构化、可执行 | 叙事式、引导式 |
| **敏感度** | 能约束行为 | 不能约束行为 |
| **例子** | 「文件位置必须是 knowledge/articles/{date}-{source}-{slug}.json」 | 「知识条目存放在 knowledge/articles/ 目录下」 |

AGENTS.md 是一份**可执行的约束文档**——它的每一条内容都直接参与 AI 的决策过程。README.md 是一份**信息型的介绍文档**——它告诉人类这个项目是什么，但不会改变 AI 生成代码的方式。

这种区别在编码规范部分表现得最明显：

在 README 里写「我们遵循 PEP 8」→ 人类读了，理解了，写代码时自己注意。
在 AGENTS.md 里写「遵循 PEP 8」→ AI 在生成每一行代码前，这条规范就在上下文中，直接影响输出。

## 一个写了 20 分钟后后悔没早做的事

如果你现在的项目没有 AGENTS.md，花 20 分钟写一份。五层结构——概述、角色、约束、指标、边界——每层 2-4 句话就够了。不需要长篇大论。

你会发现在之后的 10 次对话中，AI 的回答质量会明显提升——不是微小的提升，而是从「我每次都得纠正它」到「它主动做对了 80% 的事情」的提升。

这种提升的来源不是 AI 变聪明了。是它终于知道了你在做什么。

**AGENTS.md 不是给 AI 的说明书，是给你自己省力的工具。每一条写在里面的规范，都等于你少说了一次「不对，应该是这样」。**

# 写信 vs 对话：从一次裸 API 与 OpenCode 的对比实验，看懂 Agent 的核心抽象 (2025-07-12)

## 缘起：一封永远寄给陌生人的信

如果要我用一句话概括写这篇文章的动机，那就是——**我想搞清楚，当我们在谈论「AI Agent」的时候，我们到底在谈论什么。**

这几年 Agent 框架层出不穷：LangChain、AutoGPT、OpenCode、Claude Code……它们的宣传语里塞满了「自主」、「编排」、「记忆」、「工具调用」这些词。但如果你剥开这层营销外壳，去追问一个最朴素的问题：**一个 Agent，和我直接用 `requests` 调一次大模型 API，本质区别在哪里？**

很多人的第一反应是：「Agent 会调工具啊」「Agent 有记忆啊」。这些回答都对，但都停留在现象层。我想找到那个更底层的、能一以贯之解释所有这些现象的抽象。

于是我做了一个尽可能简单的对比实验——简单到有点无聊，但恰恰是这种无聊，把核心差异逼到了台前。

先给结论，也是贯穿全文的类比：

> **裸 API 调用，像是每次都给一个陌生人写一封完整的信；而 Agent 编排，像是和一个记得你的驻场分析师持续对话。**

写信，你必须在每一封信里交代清楚你是谁、上下文是什么、你想要什么；对话，你只需要说一句「接着刚才那个继续」。

这个「写信 vs 对话」的差异，翻译成工程语言，就是本文的主角：**无状态（Stateless）vs 有状态（Stateful）**。这，才是 Agent 真正的核心抽象。

---

## 实验设定：同一个任务，两种活法

为了让对比干净，我把任务压到最小：

- **任务**：分析一段 Python 代码（一个 Fibonacci 生成函数）的功能、优缺点和改进建议
- **被测代码**：`/tmp/task2_sample.py`
- **方式一**：裸 API 调用，Python `requests` 直连 DeepSeek Chat
- **方式二**：OpenCode CLI 编排，底层走 DeepSeek V4 Pro

注意，我刻意没有引入复杂的多轮对话、多工具链。就是「读一个文件，分析一段代码」这么一件小事。越是小事，越能看清抽象的边界在哪。

---

## 方式一：裸 API —— 手写的每一封信

先看代码。这是绝大多数人接触大模型的起点：

```python
import requests, json

API_KEY = "sk-xxx"
URL = "https://api.deepseek.com/v1/chat/completions"

code = open("/tmp/task2_sample.py").read()
prompt = f"请分析以下 Python 代码的功能、优缺点和改进建议：\n\n```python\n{code}\n```"

resp = requests.post(URL, headers={
    "Authorization": f"Bearer {API_KEY}",
    "Content-Type": "application/json"
}, json={
    "model": "deepseek-chat",
    "messages": [{"role": "user", "content": prompt}],
    "max_tokens": 800
})

result = resp.json()
print(result["choices"][0]["message"]["content"])
```

这段代码能跑，而且跑得挺好。但请注意它的每一个动作，都是我这个「写信人」亲手完成的：

- 我手动 `open()` 读了文件，又手动把内容拼进了 prompt 字符串；
- 我手动构造了 HTTP 的 headers 和 body；
- 我手动指定了 `max_tokens`，并且——如果我想知道这次花了多少 token——还得自己去翻 `result["usage"]`；
- 我手动 `resp.json()` 解析，再手动一层层剥到 `choices[0]["message"]["content"]`。

而更关键的是那些我 **没写** 的东西：

- 没有 `try/except`。网络抖一下、API 返回个 429、key 过期了，这段代码直接崩给你看；
- **没有任何上下文的延续**。这次调用结束的瞬间，一切归零。服务器不记得我，我也没保存任何东西。

最后这一点是致命的。假设我看完分析，想追问一句「那你再深入分析下它的性能」——我该怎么办？我必须把 **上一轮的完整对话**（我的问题 + 模型的回答）手动塞回 `messages` 数组，再连同新问题一起重新发一遍。而这件事，我在上面的代码里 **根本没做**——因为太麻烦了。

这就是无状态的真相：**它不是不能有记忆，而是记忆这件事的全部成本，都被甩给了调用方。**

## 方式二：OpenCode —— 一句话的对话

同样的任务，换成 OpenCode：

```bash
opencode run "请分析 /tmp/task2_sample.py 这个文件的功能、优缺点和改进建议" \
  --model deepseek/deepseek-v4-pro
```

一句话。没有 `open()`，没有 headers，没有 `messages` 数组，没有 JSON 解析。

我只是 **说了一句人话**，然后 CLI 替我完成了所有那些「写信」的脏活：

- 它 **自己识别出** 那句话里的 `/tmp/task2_sample.py` 是个文件路径，自动读取并注入 prompt——输出日志里能看到明确的一行 `→ Read /tmp/task2_sample.py`；
- 它自己构建了完整的请求；
- 它自己追踪并报告了 token 用量；
- 出错时它内置了重试；
- 它吐出来的是格式化好的 Markdown，直接能用。

更重要的是：这次对话结束后，**状态被保留了下来**。如果我接着说「再深入分析性能」，它知道「它」指的是刚才那个 Fibonacci 函数。我不需要背着上一轮的历史再走一遍——那个「背地球」的活儿，它替我干了。

## 核心差异：把两封「信」摊开对比

把两种方式的每个维度并排放，差异一目了然：

| 维度 | 裸 API（DeepSeek Chat） | OpenCode（DeepSeek V4 Pro） |
|------|------------------------|------------------------------|
| **请求构造** | 手动拼 HTTP POST、headers、body | 一句话，CLI 构建完整请求 |
| **上下文** | 无，每次调用重复传入全部上下文 | 对话历史自动保留为上下文 |
| **文件读取** | 手动 `open()` + 手动拼进 prompt | 自动识别路径、读取、注入（日志可见） |
| **状态管理** | 无，每次调用完全独立、原子 | 全自动，新对话继承历史 |
| **token 用量** | 手动翻 `usage` 字段追踪 | CLI 自动追踪并报告 |
| **错误处理** | 自己写 try/except 兜网络与 API 错误 | 内置重试与错误提示 |
| **输出** | 原始 JSON，需层层解析 | 格式化 Markdown，直接可用 |

如果你只看这张表，很容易得出一个偷懒的结论：「哦，OpenCode 就是把裸 API 封装了一下，加了点语法糖。」

**但这个结论是错的。** 表格里有六行是「便利性」的差异——请求构造、文件读取、token、错误处理、输出格式，这些确实是封装。但有一行不是，它是 **本质** 的差异：

**上下文 / 状态管理。**

其余五行，你都可以自己写个 utils 库补齐；唯独「状态」这一行，它不是一个功能，而是一种 **架构范式的转变**。

## 无状态 vs 有状态：Agent 真正的分水岭

让我把话说透。**Agent 之所以是 Agent，不是因为它会调工具，而是因为它维持了一个跨越多次调用的状态。**

大模型的 API，本质上是一个 **纯函数**：`f(prompt) → completion`。同样的输入，给出（近似）同样的输出，不依赖也不改变任何外部状态。这是无状态的定义，也是它能水平扩展、能被 CDN 缓存、能被任意负载均衡的原因。工程上，无状态是一种美德。

但「智能体」恰恰需要的是无状态的反面。一个真正在帮你干活的助手，必须记得：

- 你刚才让它看的是哪个文件；
- 它上一步得出了什么结论；
- 你们的目标是什么，进行到哪一步了。

**Agent 框架的全部工程价值，可以浓缩成一句话：它在一个无状态的 LLM 之上，构建并维护了一层状态。** 对话历史的累积、文件内容的注入、工具调用结果的回填、token 预算的追踪——所有这些琐碎功能，服务的都是同一个目标：**让一次次原子的、无记忆的 API 调用，在使用者的体感里，连成一场有记忆的、持续的协作。**

这就是为什么裸 API 让你感觉像 **接线员**——你得亲手接通每一路信号、转接每一段上下文；而 OpenCode 让你感觉像 **指挥官**——你只下达意图，执行的连续性由系统保证。

回到那个类比：

- **无状态**，是你每次沟通都要把整个地球背在身上——你是谁、背景是什么、代码全文、想要的格式，一样不能少地重新交代一遍；
- **有状态**，是你只要说一句 **「继续」** 就够了。

> **无状态是你每次都要把地球背在身上；有状态是你只要说「继续」就够了。**

这句话，是我做完这个实验最大的收获。它也解释了为什么当你尝试用裸 API 手搓一个多轮 Agent 时，你写的绝大部分代码，最后都在干同一件事：**手动维护那个 `messages` 数组。** 你是在亲手实现一个简陋的状态层。而 Agent 框架，就是那个替你把这件事做对、做全、做得可扩展的东西。

## 附：输出质量对比

顺带说下两边的产出质量。裸 API 用的是 DeepSeek Chat，OpenCode 底层走的是能力更强的 DeepSeek V4 Pro，所以这一栏严格说不是纯粹的「封装 vs 裸调」对比，而是掺入了模型代差。但结论依然有参考价值：

| 项目 | 裸 API（DeepSeek Chat） | OpenCode（DeepSeek V4 Pro） |
|------|------------------------|------------------------------|
| 功能分析 | ✓ 详细列出 | ✓ 结构化表格 |
| 优缺点 | ✓ 5 条缺点 | ✓ 8 项分类建议 |
| 改进代码 | ✓ 4 个版本（生成器、矩阵幂等） | ✓ 表格式快速建议 |
| 展示形式 | 纯 Markdown 段落 | 按类别分组的 Markdown 表格 |
| 人类操作步骤 | 写脚本 → 填 key → 运行 → 解析输出 | 一句话命令 |
| 总耗时（含思考） | ~6 秒 API + 5 分钟写脚本 | ~10 秒 CLI |

有意思的是，裸 API 那一侧反而给出了 **4 个改进版本（含矩阵快速幂）**，内容深度并不逊色——这提醒我们，**输出质量主要取决于模型本身，而编排框架的价值不在「答得更好」，而在「让你更省力地问、更连续地追问」。**

真正拉开差距的是最后两行：一边是「写脚本 → 填 key → 运行 → 解析」，5 分钟的工程开销都花在了「写信封」上；另一边是一句话，10 秒钟，精力全花在「任务本身」。

## 结语：我们真正在抽象掉的，是「重新自我介绍」的成本

绕了一圈，回到开头那封信。

裸 API 时代，我们和模型的每一次交互，都是一封 **寄给陌生人的信**：郑重、完整、自给自足，但也孤立、健忘、无法延续。你写得越多，越会发现自己 90% 的笔墨都花在了重复的自我介绍上。

Agent 编排做的事，说穿了朴素得可爱——**它记住了你**。于是那封信变成了对话，那个陌生人变成了驻场的同事，你终于可以只说「继续」，而不必每次都把地球重新背上肩。

所以，当你下次评估一个 Agent 框架时，别只盯着它支持多少工具、接了多少模型。问一个更本质的问题：**它是如何管理状态的？** 它的记忆边界在哪，上下文怎么积累，又在何时被清空？

因为归根结底，**Agent 工程的深水区，从来不是让模型答得更聪明，而是让协作变得连续。** 无状态到有状态这一步跨过去，你才算真正从「调用一个 API」，走到了「拥有一个 Agent」。

从写信，到对话。这一步，值得每一个 Agent 开发者认真对待。

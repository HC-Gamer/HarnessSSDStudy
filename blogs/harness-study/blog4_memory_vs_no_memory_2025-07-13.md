# 给 AI 一本「员工手册」：AGENTS.md 如何驯服代码生成的随机性 (2025-07-13)

## 为什么需要「员工手册」

想象你第一天入职一家新公司。没人告诉你代码风格是什么、Commit 规范怎么写、CI 流程怎么跑。你按自己习惯写——函数名 camelCase、注释看心情、错误处理写到哪里算哪里。这能跑，但三个月后同事接手你的模块，会骂人。

AI 编程工具面临一模一样的困境。每次你打开一个新的对话，AI 都是"第一天入职"——它不知道你项目的编码规范、命名风格、日志策略、架构约定。它按"出厂设置"回应你：函数名混用 camelCase 和 snake_case、docstring 写得像朋友圈、错误处理一把 `try/catch` 兜底。

这能跑。但在一个超过 5 个文件的项目里，这种随机性就是技术债务的开端。

AGENTS.md（Claude Code 叫 CLAUDE.md、Cursor 叫 .cursorrules）就是解决这个问题的方案。它不是复杂的技术，就是一个 Markdown 文件，放在项目根目录。但它做了一件重要的事：**在所有 AI 做决策的"模糊时刻"提供方向指引**。

## 实验设计：同一句话，两个世界

为了验证 Memory 文件的实际效果，我们设计了一个简单的对比实验。

### 实验步骤

1. **有 Memory 组**：项目根目录存在 AGENTS.md，定义项目规范（Python 技术栈、snake_case 命名、Google 风格 docstring、禁止裸 print()、类型注解要求等）。向 AI 提问：
   > "帮我写一个 Python 函数，实现从 GitHub API 获取指定仓库的基本信息（Star 数、Fork 数、描述），保存为 utils/github_api.py"

2. **无 Memory 组**：临时移除 AGENTS.md。向 AI 提问**完全相同的问题**，保存为不同的文件名。

3. **对比**：从 9 个维度逐项比较两个文件的差异。

### 结果：差距大到不需要统计检验

| 检查项 | ✅ 有 Memory | ❌ 无 Memory |
|--------|------------|-------------|
| 命名风格 | snake_case（`get_repo_info`） | camelCase（`getGithubRepoInfo`） |
| 文档字符串 | Google 风格（Args/Returns/Raises） | 随意风格，像代码注释 |
| 日志方式 | `logging` 模块，可配置级别 | 裸 `print()` |
| 错误处理 | 具体异常类型 + 指数退避重试 | 一把 `except Exception` |
| 超时控制 | 明确设置 `timeout=10` | 无超时设置 |
| 返回值设计 | 结构化返回 + 异常时为 None | 返回格式不固定 |
| 类型注解 | 完整类型提示 | 无类型注解 |
| 常量管理 | 模块级常量集中定义 | 字符串硬编码 |
| 文件位置 | 按项目结构组织 | 无结构意识 |

这不是微小的差异。两个文件的代码质量差了不止一个档次——有 Memory 的版本可以直接上生产，无 Memory 的版本上线前需要大量重构。

## Memory 是如何工作的？

"员工手册"这个比喻之所以贴切，是因为 AGENTS.md 的运作方式和公司的员工手册几乎一样：

**员工手册不会告诉你这个 Bug 怎么修，但它告诉你代码审查的标准是什么。AGENTS.md 不会替 AI 写每一行代码，但它在每一个"模糊时刻"提供方向。**

具体来说，AGENTS.md 中的每一条规则，在 AI 生成代码时扮演一个"拦截器"的角色：

```
AI 想：这个函数叫什么？
  → AGENTS.md 说「snake_case」
  → 锁定：get_repo_info

AI 想：要不要写文档？
  → AGENTS.md 说「所有公开函数必须有 docstring（Google 风格）」
  → 锁定：完整的 Args/Returns/Raises 三段式

AI 想：出错了怎么处理？
  → AGENTS.md 说「禁止裸 print()，使用 logging」
  → 锁定：logging.error + 重试机制
```

这个过程不是显式的 if-else 判断——AI 不会在每次决策时翻开 AGENTS.md 逐行查找。更好的理解方式是：**AGENTS.md 的内容被注入到 AI 的上下文窗口中，作为系统提示的一部分**。它改变了 AI 对"这个项目是什么风格"的先验认知，使得后续的所有生成动作都自动对齐到规范上。

这也是为什么 Memory 文件的措辞风格很重要——你是在给 AI 写规范，不是在给人类写文档。人类能容忍模棱两可的表述（"代码风格统一即可"），AI 需要可执行的定义（"Python 变量使用 snake_case，类名使用 PascalCase"）。

## 一个常见的误解

有人说："我每次对话都会告诉 AI 我的规范啊。"

这确实是一个替代方案——每次对话开头写一段 prompt："请使用 snake_case，Google 风格 docstring，用 logging 代替 print。"问题在于：

1. **遗忘成本**：10 句话的对话里提一次，50 句话的对话里可能已经忘了 80% 的约束。
2. **不一致性**：今天你说了 A，明天你说了 B，AI 生成的文件风格不统一。
3. **上下文碎片**：每次重复的"规范提示"占用了宝贵的上下文窗口。

AGENTS.md 的价值恰恰在于它**一次声明，到处生效**。只要文件在项目根目录，AI 每次启动时自动读取，根本不需要你口头交代。

## 什么应该写在 Memory 文件中

基于实验经验，我建议 AGENTS.md 包含以下五类内容：

**1. 项目定位（1-2 句话）**

让 AI 知道它在这个项目中扮演什么角色。不是技术描述，而是意图描述。比如"这是一个全自动的知识管理系统，每天从 GitHub Trending 和 Hacker News 采集 AI 相关内容"——这告诉了 AI 它在做"自动化采集"而不是"手动查询"。

**2. 技术栈声明**

语言版本、依赖管理方式、必须使用的框架。这能防止 AI 在一个 Python 3.8 项目里生成 match-case 语法，或者在一个 Poetry 项目里输出 pip freeze。

**3. 编码规范（最重要的部分）**

这是消除随机性的核心。需要明确：
- 命名风格（snake_case / camelCase / PascalCase）
- 文档规范（Google / NumPy / reStructuredText）
- 日志策略（logging 框架 vs print）
- 错误处理模式（重试策略、异常类型）
- 类型注解要求
- 文件编码

**4. 项目结构**

告诉 AI 文件应该放在哪里。Utils 放哪里？测试放哪里？配置放哪里？这能防止 AI 在根目录下一口气生成 10 个文件。

**5. 红线**

什么绝对不能做。这在安全场景尤其重要——"不在日志中输出 API Key"、"不修改 AGENTS.md 本身"。

## 从数字看差距

有 Memory 的文件：101 行代码，7 个函数/常量，完整测试路径覆盖。
无 Memory 的文件：45 行代码，1 个函数，无类型注解，无重试机制。

这不是"好"和"更好"的区别，这是"可维护"和"需要重写"的区别。

## 你学到了什么

AGENTS.md 不是银弹——它不能替代好的架构设计，不能修复混乱的业务逻辑。但它是 AI Agent 工程中**成本最低、收益最高的**实践之一。

创建它只需要 10 分钟，写好 20 行规范。但它会改变之后每一次对话的产出质量。当你的项目从 3 个文件长到 30 个文件、从 1 个开发者变成 3 个 AI Agent 协作的时候，你会发现这一点：

**没有 Memory 的项目，AI 每次都是新人。有 Memory 的项目，AI 是知道规范的同事。**

这就是「声明式配置」的威力——用一个文件声明"要什么"，不用每次口头交代。

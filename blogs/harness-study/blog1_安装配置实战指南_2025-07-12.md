# OpenCode 安装与配置实战指南：给 Agent 开发者的一把趁手工具 (2025-07-12)

> 如果你写过一次「裸调 LLM API + 手动解析工具调用 + 手动维护上下文」的循环，你就会明白 OpenCode 这类框架存在的意义。它不是又一个 CLI 玩具，而是把 Agent 编排里那套脏活累活替你封装好的框架。

---

## 一、先说「为什么」：Agent 开发者到底在痛什么

在讲怎么装、怎么配之前，我想先花点篇幅聊聊「为什么」。因为如果你只是想要一个「能在终端里跟 AI 聊天写代码」的工具，那市面上一抓一大把；但如果你是想理解 **Agent 是怎么跑起来的**、甚至想自己搭一套，那 OpenCode 的价值就完全不同了。

我们回忆一下，自己动手撸一个「能读写文件、能跑命令」的 Agent，要处理哪些事：

1. **HTTP 请求的样板代码**。每个模型厂商的 endpoint、鉴权头、请求体格式都略有差异。DeepSeek 是 OpenAI 兼容的，Qwen 挂在 dashscope 的 compatible-mode 下，Kimi 又是另一个域名。你要么给每家写一套 client，要么自己抽象一层适配器。

2. **上下文管理**。多轮对话要拼 message 数组，长了要截断或摘要，还要算 token。这部分逻辑写起来不难，但写对、写稳、写到能上生产，是另一回事。

3. **工具调用（tool calling）的解析与回填**。这是最折磨人的地方。LLM 返回一个「我要调用 `read_file` 工具，参数是 xxx」的结构化响应，你得解析它、真的去执行、再把执行结果塞回对话里、然后**再发一次请求**让模型继续。这个「请求—解析—执行—回填—再请求」的循环，就是 Agent 的心脏。

4. **流式响应的处理**。用户不想盯着转圈，你得处理 SSE 流，一边收一边渲染。

这四件事，每一件单独看都不算难，但**组合在一起、并且要做得健壮**，就是一个不小的工程量。更别提你可能今天想用 DeepSeek、明天想切 Claude、后天老板要求接私有化部署的模型——每换一次后端，上面四件事有一半要重写。

这就是 OpenCode 要解决的问题。

---

## 二、OpenCode 的定位：它是编排框架，不是聊天客户端

我给 OpenCode 的一句话定位是：

> **一个终端里的 Agent 编排框架，把「请求—解析工具—执行—回填」的循环做成了开箱即用的基础设施，并且对任意 OpenAI 兼容 API 保持开放。**

注意「编排框架」这四个字。它和「聊天客户端」的本质区别在于：聊天客户端只负责把你的话发给模型、把回复显示出来；而编排框架要负责**让模型去操作真实世界**——读你的文件、改你的代码、跑你的命令，并且在一个循环里自主决策「下一步该干嘛」。

我们把它的运行逻辑压缩成一张图：

```
用户输入 → OpenCode CLI → 读取系统 prompt（含 tool 定义）
  → 调用 LLM API → 解析流式响应
    → 若 LLM 要求用工具 → 执行工具（读写文件/跑命令）→ 结果喂回 LLM
    → 若 LLM 认为已完成 → 输出结果
```

这个循环的精髓，在于最后那个**二选一的决策**：模型每转一圈，都在回答同一个问题——「我是该继续调工具，还是我已经答完了？」。它自己判断、自己收敛，直到任务真正完成。你写业务逻辑时，不需要关心这一圈圈是怎么转的，这正是框架帮你扛下来的部分。

理解了这一点，你就明白为什么我说它对 **Agent 学习者** 特别友好：它把一个教科书级别的 Agent Loop 实现摆在你面前，你可以直接用，也可以读它、拆它、仿它。

## 三、安装：三行命令的事

安装本身没有任何惊喜，这是好事——基础设施就该无聊。

### macOS（推荐 Homebrew）

```bash
brew install opencode
```

### 其他平台 / 想要全局命令（npm）

```bash
npm install -g @opencodeai/opencode
```

### 验证

```bash
opencode --version
# 输出示例: 1.17.10
```

两种方式我都试过，如果你已经在用 Homebrew 管理开发工具链，首选 brew，升级更省心；跨平台或者 CI 环境里，npm 更通用。

## 四、配置国产模型：重点在 `opencode.json` 与 `secrets.env` 的配合

这是全文的核心，也是我认为 OpenCode 设计得最漂亮的地方。它没有为每个厂商写一堆硬编码的 provider，而是通过 `@ai-sdk/openai-compatible` 这个适配器，把「接入任意 OpenAI 兼容 API」简化成了**改一个 `baseURL`**。

对国内开发者来说这太重要了——DeepSeek、Qwen、GLM、Kimi 全都提供 OpenAI 兼容接口，意味着接入成本几乎为零。

### 4.1 建配置目录

```bash
mkdir -p ~/.config/opencode
```

### 4.2 主配置 `~/.config/opencode/opencode.json`

以 DeepSeek 为例：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "deepseek": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "DeepSeek",
      "options": {
        "baseURL": "https://api.deepseek.com",
        "apiKey": "{env:DEEPSEEK_API_KEY}"
      },
      "models": {
        "deepseek-v4-pro": { "name": "DeepSeek V4 Pro" },
        "deepseek-v4-flash": { "name": "DeepSeek V4 Flash" },
        "deepseek-chat": { "name": "DeepSeek Chat" },
        "deepseek-reasoner": { "name": "DeepSeek Reasoner" }
      }
    }
  },
  "model": "deepseek/deepseek-v4-pro"
}
```

这里有几个细节值得点出来：

- **`"$schema"` 别删**。它给你的编辑器提供了自动补全和校验，配错字段会当场报红，能省下大量「为什么不生效」的排查时间。
- **`apiKey` 用的是 `{env:DEEPSEEK_API_KEY}` 占位符，而不是明文**。这是我强烈推荐的做法，原因下一节讲。
- **顶层 `"model"`** 指定默认模型，格式是 `provider/model`，即 `deepseek/deepseek-v4-pro`。命令行不带 `--model` 时就用它。

### 4.3 密钥文件 `~/.config/opencode/secrets.env`

```
DEEPSEEK_API_KEY=sk-你的key
```

> OpenCode 启动时会自动 source 这个文件，所以配置里的 `{env:DEEPSEEK_API_KEY}` 能读到值。

**为什么要把 key 单独拆出来，而不是直接写进 `opencode.json`？** 这是一个看起来微不足道、实则关乎工程卫生的决定：

1. **可分享配置，不泄露密钥**。`opencode.json` 里全是结构化配置，你可以放心地丢进团队仓库、发到博客、贴给同事；`secrets.env` 则永远躺在本地、进 `.gitignore`。配置与密钥的关注点分离，是最基本的安全素养。
2. **多环境切换方便**。开发、测试、不同额度的 key，换一个 `secrets.env` 就切换了，主配置纹丝不动。

如果你在这一步图省事把 key 硬编码进 json，大概率会在某次 `git add .` 之后追悔莫及。别问我怎么知道的。

## 五、其他国产模型：格式完全一致，只改两处

这就是「改一个 `baseURL`」的威力。想接别家，在 `provider` 里加一个同结构的对象，`secrets.env` 里加一个对应变量，完事。

| 模型 | `baseURL` | 适配器 |
|------|-----------|--------|
| **阿里 Qwen** | `https://dashscope.aliyuncs.com/compatible-mode/v1` | `@ai-sdk/openai-compatible` |
| **智谱 GLM** | `https://open.bigmodel.cn/api/paas/v4` | 同上 |
| **月之暗面 Kimi** | `https://api.moonshot.cn/v1` | 同上 |

配好几家之后，你会得到一个特别爽的能力：**同一个任务，一行命令换个模型跑一遍做对比**。哪家便宜、哪家聪明、哪家跑工具调用更稳，自己心里就有数了。

## 六、验证对话：确认整条链路通了

配置写完别急着干活，先发一句话确认端到端跑通：

```bash
DEEPSEEK_API_KEY="sk-你的key" opencode run "你好，请回复：配置成功" --model deepseek/deepseek-v4-pro
```

成功的话你会看到：

```
> build · deepseek-v4-pro
配置成功
```

这里我在命令行里显式带了一次 `DEEPSEEK_API_KEY=`，是为了首次验证时排除「环境变量没读到」这个最常见的干扰项。确认能通之后，日常使用就靠 `secrets.env` 自动加载，不用每次都手写 key。

## 七、常用命令速查

| 命令 | 说明 |
|------|------|
| `opencode run "..."` | 单次对话模式，适合脚本化、CI |
| `opencode`（无参数） | 交互式 TUI 模式，适合日常长任务协作 |
| `opencode --model deepseek/deepseek-v4-pro` | 临时指定模型，覆盖默认配置 |
| `--dangerously-skip-permissions` | 跳过工具执行的权限确认 |

关于最后那个 `--dangerously-skip-permissions`，官方把「dangerously」写进 flag 名里，是有诚意的警告。默认情况下，OpenCode 在**真正动你的文件系统或跑命令之前会请求确认**——这是一道重要的安全闸门。跳过它意味着 Agent 可以无阻拦地读写、执行。我的建议是：

- **探索、学习、跑在受控目录里**：可以开，效率高。
- **在有真实代码、真实数据的仓库里**：老老实实一步步确认，尤其是当你用的模型你还不够信任的时候。一个自主循环里的 Agent，能力越大，失手的破坏也越大。

## 八、和「裸调 API」的体验差，到底差在哪

写到这我想做个诚实的对比，因为总有人会问：「我自己 `curl` 或者写几十行 Python 不也能调？」

能，但你调的是**模型**，不是**Agent**。差别在于：

| 维度 | 裸调 API | OpenCode |
|------|----------|----------|
| 工具调用循环 | 自己写「解析—执行—回填—再请求」| 框架内置，开箱即用 |
| 多后端切换 | 每家改 client / 改请求格式 | 改一个 `baseURL` |
| 上下文维护 | 手动拼 message、手动截断 | 框架托管 |
| 文件/命令操作 | 自己实现工具 + 权限控制 | 内置工具 + 权限闸门 |
| 流式渲染 | 自己处理 SSE | 终端里直接跑 |

一句话：裸调 API 你得到的是「一次问答」；OpenCode 给你的是「一个能自己转很多圈、直到把活干完的循环」。前者是零件，后者是引擎。

当然，这不是说裸调没意义——如果你要**理解** Agent 的原理，亲手写一遍那个循环是最好的学习方式，我强烈建议每个 Agent 开发者都做一次。但当你要**用** Agent 干活时，没必要每次重造轮子。理解原理靠手写，提升效率靠框架，两者不矛盾。

## 九、个人推荐与几点注意

聊到最后，给几条我自己踩过坑之后总结的建议：

1. **配置与密钥永远分离**。`opencode.json` 可公开，`secrets.env` 进 `.gitignore`。这条我重复第三遍，因为它值得。
2. **善用 `$schema`**。让编辑器帮你校验，别用肉眼 debug JSON。
3. **首次验证时显式传 key**，排除环境变量问题；跑通后再依赖自动加载。
4. **权限闸门默认开着，别急着 skip**。把 `--dangerously-skip-permissions` 当成「我完全清楚接下来会发生什么」时才用的开关。
5. **配好多家模型做横向对比**。国产模型迭代很快，别锁死在一家，用一行命令的成本去感受差异，判断留给数据。
6. **把它当学习材料**。如果你在学 Agent，OpenCode 那个「继续用工具 vs 我答完了」的编排循环，本身就是一份活的教材——它跑给你看的每一步，都是你将来自己写 Agent 时要处理的每一步。

---

## 尾声

工具的价值，从来不在于它有多花哨，而在于它替你消灭了多少重复的痛苦。OpenCode 消灭的，正是每个 Agent 开发者都绕不过去的那个循环——请求、解析、执行、回填，一圈又一圈，直到模型自己说「我答完了」。

把这个循环交给框架，把你的注意力留给真正重要的事：你想让 Agent 替你完成什么。

装上，配好，跑起来。剩下的，交给那个循环。

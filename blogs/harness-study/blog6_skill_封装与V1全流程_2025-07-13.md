# Skill 封装 + V1 全流程：从零到知识条目的 3 步管线 (2025-07-13)

## 问题的症结：AI 不知道什么时候该出手

在单 Agent 场景下，AI 的行为是完全随机的——你问它"帮忙看一下这几个仓库"，它可能给你一段自然语言描述。你问同一个问题第二遍，它给你表格。第三遍，它直接写代码。

这种不一致不是因为 AI 能力不够，而是因为**你没有告诉它固定的操作流程**。

Skill 文件就是干这个的：把"采集 GitHub Trending"变成一个有 7 个步骤的 SOP，AI 只要命中 description 里描述的场景，就自动按步骤执行。

## 什么是 Skill：从招聘广告到岗位说明书

Agent 和 Skill 的区别可以类比到人类团队：

| | Agent（角色） | Skill（能力） |
|------|--------------|---------------|
| 回答的问题 | "我是谁？" | "怎么干？" |
| 类比 | 岗位说明书（招聘广告） | 操作手册（SOP） |

Agent 定义了谁来做、有什么权限；Skill 定义了做什么、按什么顺序做、输出什么格式。

## github-trending Skill：从采集到归档的 7 步

最让人头疼的问题是 Skill 的 description 字段怎么写得恰到好处。太宽泛了——AI 在不相干的时候触发它。太窄了——AI 该用的时候不用。

B 路的做法是先用 spec 设计 description：

```yaml
name: github-trending
description: >
  当需要采集 GitHub 热门开源项目、获取 GitHub Trending 数据、
  搜索 AI 仓库时使用此技能。适用于知识库采集阶段。
```

覆盖了三种自然语言表达方式：
- "采集热门项目" → 触发
- "获取 GitHub Trending" → 触发
- "搜索 AI 仓库" → 触发

更重要的是 allowed-tools：

```yaml
allowed-tools:
  - Read
  - Grep
  - Glob
  - WebFetch
```

这是对 Agent 权限的进一步细化。即使 Collector Agent 有 WebFetch 权限，If github-trending skill 不在 allowed-tools 里声明 WebFetch，AI 仍然不能调用。**Skill 可以进一步收紧 Agent 的权限。**

7 步执行流程：

1. **搜索**：GitHub API 搜索近 7 天创建、star > 100 的仓库
2. **提取**：name, url, stars, language, topics
3. **过滤**：纳入 AI/LLM/Agent 相关，排除 Awesome 列表
4. **去重**：按 full_name 严格去重
5. **摘要**：中文，公式「项目名 + 做什么 + 为什么值得关注」
6. **排序**：按 star 降序取 Top 15
7. **输出**：JSON 到 `knowledge/raw/github-trending-YYYY-MM-DD.json`

每一步的执行标准都写死了。AI 没有自由发挥的空间。

## tech-summary Skill：打分需要约束

分析 Skill 最难做的是评分约束。AI 天然倾向于给高评分——如果你不做硬性限制，15 条数据它会打出 8 个 9 分。

解决方法是在 Skill 里写死评分约束：

```yaml
## 评分标准
| 分数 | 含义 |
| 9-10 | 改变格局 |
| 7-8 | 直接有帮助 |
| 5-6 | 值得了解 |
| 1-4 | 可略过 |

## 约束
- 15 个项目中 9-10 分不超过 2 个
```

测试时可以看到这个约束确实生效了。15 条数据中 9 分仅 2 个（openai/codex 和 ggerganov/llama.cpp），其余 13 条分布在 6-8 分。没有评分膨胀。

## V1 全流程：采集 → 分析 → 整理

当一个 Agent 拥有一组 Skill 时，完整的 V1 流程就是三步：

```
Agent + Skill 联合触发

Step 1: @collector 调用 github-trending skill
  → 输出 knowledge/raw/github-trending-YYYY-MM-DD.json

Step 2: @analyzer 调用 tech-summary skill  
  → 读取 raw 数据 → 深度分析 → stdout JSON

Step 3: @organizer 整理归档
  → 读取分析结果 → URL hash 去重 → 每条独立 JSON
  → 存入 knowledge/articles/{date}-{source}-{slug}.json
```

### 验证结果

**Step 1 — 采集（15 条）**
- 来源：GitHub Search API
- 过滤规则生效：排除非 AI 项目
- 质量检查：全部有 name + url + summary（中文）

**Step 2 — 分析（15 条）**
- 摘要：<= 50 字，全部中文
- 亮点：每条 2-3 个（含具体数据）
- 评分：6-9 分，区分度良好
- 约束执行：9 分仅 2 个（硬性约束生效）

**Step 3 — 归档（15 个文件）**
- 去重：URL hash 去重正常
- 格式：标准 JSON，所有字段完整
- 状态说明：全部 mark 为 draft（待审核）

### 发现：V1 的三个局限

跑完 V1 后发现了三个问题，留给后续迭代：

1. **上游失败处理**：collector 失败时，analyzer 仍然尝试读取最新文件。目前的容错是"失败时返回空数组"——但下游不知道上游是"采集成功但结果为 0"还是"采集失败"。

2. **没有运行时监控**：三步串行执行，没有心跳/日志。如果某一步卡住超过 30 秒，没有机制被动检测出来。

3. **数据格式耦合**：analyzer 预期的 items 结构必须在 collector 的 schema 中严格匹配。一旦 collector 改了输出格式，analyzer 和 organizer 都会断。

## 四要素总结

V1 架构的本质是四个部分：

- **Memory**（AGENTS.md）= 项目规范和行为约束
- **Sub-Agents**（.opencode/agents/*.md）= 角色定义和权限矩阵
- **Skills**（.opencode/skills/*/SKILL.md）= 可执行的操作手册
- **知识数据**（knowledge/）= 所有产出的最终归宿

这三层——Agent 定义角色，Skill 定义步骤，数据定义输出——构成了一个可重复、可审计、可迭代的知识采集系统。每一步都有文件，每一步都有标准，每一步都知道下一步该做什么。

这是 SDD 赋予你的确定性——不是让 AI 更聪明，而是让 AI 的行为可预测。
